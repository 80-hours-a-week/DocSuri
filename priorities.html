01 · Framework

평가 프레임워크

다섯 축을 1-5점으로 평가, 가중합으로 35점 만점. 의존성은 별도 그래프로 강제.

V
Value
사용자가 얼마나 자주·크게 가치를 얻는가
×2.0
D
Differentiation
시장 일반 도구 대비 차별성
×1.5
C
Cost-eff.
단위 비용 + 캐시·재사용 (낮을수록 5점)
×1.0
R
Risk (역)
환각·법무·정책 위험 (낮을수록 5점)
×1.0
E
Effort (역)
신규 인프라·외부 의존성 (적을수록 5점)
×1.0
🧮
총점 공식
Total = V × 2 + D × 1.5 + C + R + E   (만점 35)
Value·Differentiation에 가중치를 부여 — 비용·리스크·노력은 의사결정의 보조 요인이지 결정 요인이 아니라는 판단.

07 · Notable Decisions

주목할 만한 결정 3가지

1
왜 #11·#09를 #03보다 앞에 두는가
단순 가치 점수상 #03(번역)이 더 높지만 — 빌드 순서는 다름
두 기능 모두 단가가 한 자릿수 센트($0.06·$0.09)에 차별화 5점. spec 본문이 명시한 "최저 단가 + 최고 lock-in"(#11) 과 "rubric 투명성 + 저자 prereview"(#09) 가치를 빠르게 풀어 사용자 신뢰를 만드는 게 비싼 후속 기능보다 ROI가 큼. 번역은 가치는 높지만 인프라가 #02와 강하게 겹치므로 묶어서 같은 분기에 처리하는 게 효율적.
2
왜 #06이 V=4 D=4인데도 10위인가
"공백" 자체가 검증 불가능한 negative claim
단위 $12 (다른 기능의 100~200배) + "negative claim 검증 본질적 불가능" 위험이 spec에 명시. spec 본문이 직접 "환각 위험 최상위", "출판사 IP 분쟁", "사용자 입력 격리 필수"를 언급할 정도라면, 다른 기능으로 anchor·verifier·세션 격리 패턴이 충분히 검증된 뒤에 들어가야 안전.
3
왜 #10이 마지막인가 — 정책이 아닌 의존성 때문
AGENTS.md 폐기로 정책 충돌은 해소됨
spec 본문에 "Literature Map = 06+05 재사용", "Citation Candidate = 임베딩 ANN", "Peer Review = rubric (09 패턴 공유)" 가 명시. 이는 #10이 자체 기능이라기보다 다른 기능의 합성 UI라는 뜻 — 구성 요소가 모두 안정된 뒤에야 합치는 게 자연스러움.

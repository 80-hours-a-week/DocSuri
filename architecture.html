동기 응답

검색 결과와 이미 캐싱된 요약·번역은 즉시 응답합니다. 외부 API 병렬 호출도 모두 동기 라우터 안에서 처리합니다.

비동기 큐

PDF 파싱과 임베딩처럼 수십 초 걸리는 작업, 그리고 LLM 스트리밍 호출은 Celery로 분리했습니다. 라우터는 작업 등록 직후 202를 응답합니다.

캐시 우선

검색은 15분 TTL로 캐싱하고, 요약과 번역은 영속 캐시로 보관합니다. 같은 논문을 다시 요청하면 LLM 호출 없이 결과만 돌려줍니다.

💡 구조 고정의 이점
5개 필드가 항상 같은 스키마이기 때문에, 서로 다른 논문 간에 '방법 vs 방법'을 사이드-바이-사이드로 비교할 수 있습니다. JSON 모드를 사용하므로 파싱 실패 위험도 사실상 사라집니다.

04 · Orchestration Layer
v1.1 추가

LangGraph 워크플로 설계

각 워크플로는 상태 그래프로 선언합니다. 노드별로 체크포인트를 저장하기 때문에 중간에 실패해도 그 노드부터 재개할 수 있습니다 — Celery 재시도가 task를 처음부터 다시 도는 것과는 결정적으로 다릅니다.

공유 패턴 · State Schema

TypedDict + reducer
class SummaryState(TypedDict):
  paper_id: str
  lang: Literal["ko", "en"]
  sections_text: str
  raw_response: str | None
  validated: SummarySchema | None
  retry_count: Annotated[int, operator.add]
  errors: Annotated[list[str], operator.add]
  tokens_in: int
  tokens_out: int
Annotated[..., operator.add]는 LangGraph의 reducer 어노테이션입니다. 여러 노드가 같은 필드를 갱신할 때 마지막 값으로 덮어쓰지 않고 누적하도록 만듭니다. 재시도 횟수나 에러 로그처럼 발생 이력을 모두 보존해야 하는 필드에 사용합니다.
CHECKPOINTER
PostgresSaver
상태 스냅샷을 Postgres의 checkpoints 테이블에 저장합니다. thread_id 단위로 그룹화하므로, 같은 요약 요청에서 발생하는 스냅샷들은 같은 thread에 묶입니다.
Resume 시나리오
번역 청크 10개 중 5개를 처리한 시점에 Claude rate limit에 걸렸다고 가정합시다. 다음 워커가 픽업해 graph.invoke(state, thread_id=...)를 호출하면 6번째부터 자동으로 재개합니다. 이미 끝낸 1~5번 결과는 그대로 보존되어 있어 다시 호출할 필요가 없습니다.

Send API(LangGraph의 동적 fan-out 도우미)로 청크별 노드를 동시에 실행합니다. 용어 일관성 검증에 실패한 청크만 다시 번역하므로, 전체를 재처리할 필요가 없습니다.
WHY · 1
노드 단위 재개
Celery는 재시도 시 task 전체를 처음부터 다시 돌립니다. 반면 LangGraph는 실패한 노드부터 재개하므로, 이미 끝낸 비싼 LLM 호출 결과를 버리지 않습니다.
WHY · 2
조건부 라우팅
'검증 실패 시 재시도', '용어 충돌 시 재번역' 같은 동적 분기를 그래프 엣지로 선언적으로 표현할 수 있습니다. 코드 안의 if-else 체인보다 흐름이 한눈에 들어오고, 분기 조건을 바꾸기도 쉽습니다.
WHY · 3
관찰성과 합
LangSmith나 Langfuse가 각 노드를 자동으로 OpenTelemetry span으로 추적합니다. 어느 노드가 느리고 어디서 실패하는지를 코드 변경 없이 바로 확인할 수 있습니다.

07 · Non-Functional Requirements

비기능 요구사항

CATEGORY 처리 방식
지연 (요약) SSE 스트리밍으로 첫 토큰까지 약 2초, 5개 섹션 완료까지 약 15초를 목표로 합니다.
지연 (번역) 청크를 병렬로 호출하여 A4 한 쪽 분량을 약 3초에 처리합니다.
비용 Prompt Caching, Haiku 번역, 요약 캐시 영속화를 결합해, 동일 논문을 다시 요청할 때 LLM 호출 비용을 0원으로 만듭니다.
재시도 Celery의 autoretry_for와 지수 백오프(exponential backoff)를 함께 사용합니다.
레이트 리밋 arXiv는 요청당 3초 간격, Semantic Scholar는 5분당 100건 — Redis 토큰 버킷으로 강제합니다.
저작권 arXiv는 오픈 액세스라 본문 저장이 가능하지만, Semantic Scholar는 본문 저장이 금지되어 있어 메타데이터만 저장합니다.
관찰성 LLM trace는 Langfuse, API 메트릭은 Prometheus가 담당합니다.

RULE · 1
Mock-first 통합
첫 단계에는 모두가 인접 기능에 mock 응답을 먼저 제공합니다. 예를 들어 F2·F3는 F1의 실제 PDF가 없어도 하드코드 텍스트로 그래프를 검증할 수 있고, F1은 F4의 라우터가 없어도 자체 mock 라우트로 검색 UI를 띄울 수 있습니다.
RULE · 2
계약은 PR로
핸드오프 매트릭스의 인터페이스(Section 모델, checkpointer 규약, 캐시 키, 디자인 토큰)는 변경할 때 반드시 PR 리뷰를 거칩니다. 기능 내부 구현은 자유롭게 바꾸어도 되지만, 다른 기능과의 계약 변경에는 양쪽의 합의가 필요합니다.
RULE · 3
관찰성은 F4가 깔되
지표는 각 기능이
Langfuse와 OTel 인프라는 F4가 깔아 둡니다. 그 위에서 각 기능 팀이 자기 지표를 직접 계측합니다 — F1은 외부 API 지연·실패, F2와 F3는 각자의 LLM 토큰·비용과 자기 UI의 클라이언트 지연을 담당합니다.

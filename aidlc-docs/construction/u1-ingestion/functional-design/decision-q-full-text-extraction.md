## 0. 배경 (플래그 — 문서·코드 불일치)

- U1 기능설계 **Q2=A**(`u1-ingestion-functional-design-plan.md:78`)는 데모 범위를 **"초록+메타데이터만, 전문 본문 제외"**로 확정했다. 그 결과 "전문 취득/파싱/OA 전문 보관은 부재(단순화)"가 의식적으로 수용됨.
- 그러나 이후 코드에 전문 취득·보관이 **추가**되었다: `arxiv.py:67` `fetch_full_text` → `application.py:131` `put_full_text` → `aws.py:27` `S3FullTextStore`. U7 요약은 이 S3 전문을 **소스로 전제**한다(U7 Q1: "요약=S3 전문").
- 즉 **전문 추출은 "어떻게 읽을지"가 설계된 적 없이 붙었다.** 이 누락이 #139의 근원.

## 1. 문제 (관측 — 실측)

`arxiv.py:69`는 전문을 **arXiv e-print 엔드포인트**(`https://arxiv.org/e-print/{id}`)에서 받는다. 이 응답은 PDF/플레인텍스트가 아니라 **gzip으로 압축된 tar(LaTeX 소스 묶음)**다. 그런데 `arxiv.py:73`은 **압축 해제 없이** `body.decode("utf-8", errors="replace")`로 바로 디코딩한다.

- 실측(`1706.03762`): 응답 선두 매직바이트 `1f 8b`(gzip). 위 디코딩 시 결과의 **44.3%가 깨진 문자(�)**.
- 결과적으로 S3에 저장되는 "전문"이 사람이 읽을 수 없는 텍스트이고, 전문 뷰어·요약·번역 입력에 그대로 흘러간다.
- 대조: **에셋(그림) 추출 경로는 같은 묶음을 이미 정상 처리**한다(`asset_extraction.py:145` `tarfile.open(..., mode="r:*")`). 본문 경로만 묶음 해제를 누락.

## 2. 가정 (잘못이면 정정 요청)

- A-1. 전문 보관은 **유지**한다(U7 요약/번역/뷰어가 의존하므로 Q2=A의 "전문 부재"로 되돌리지 않는다).
- A-2. 추출 의존성은 **현재 스택 내** 우선, 단 **배포 품질에 필요하면 신규 도입을 허용**한다(데모 단순화에 묶이지 않음). 기존: `pdfplumber`/`pypdfium2`(PDF), `tarfile`(표준).
- A-3. **두 소비자의 요구가 다르다**: 뷰어는 **수식·표·구조가 살아있는 충실한 본문**, AI는 **깨끗한 평문**. 한 소스에서 둘 다 파생할 수 있으면 최선(중복 취득·불일치 회피).
- A-4. arXiv는 출판물 중 드물게 **저자 소스(LaTeX)** 를 제공 → PDF만 있는 일반 출판물보다 **더 좋은 패(구조화 HTML)** 가 존재한다. 이 이점을 살린다.

## 3. 결정 질문

**Q — 전문을 어떤 소스/방식으로 추출해 S3에 보관하나?**

- **A) e-print(LaTeX 소스) 해제 + .tex 병합.** gzip 해제 → tar에서 `.tex` 멤버 추출·병합 → 정규화 후 보관. (에셋 경로의 tar 처리 재사용)
  - 장점: 새 의존성 없음(표준 `tarfile`), 그림은 이미 같은 묶음에서 분리 중 → 일관.
  - 단점: 본문에 **LaTeX 마크업이 잔존**(`\citep{}`·수식 명령). 뷰어 가독성·요약 입력 품질이 마크업에 의해 저하. 다중 파일(`\input`) 병합 순서 처리 필요.
- **B) PDF 텍스트 추출.** e-print 대신 `arxiv.org/pdf/{id}` 취득 → `pdfplumber`로 텍스트 추출 → 정규화 후 보관.
  - 장점: **렌더 본문에 가장 근접**(마크업 없음). 의존성·PDF 바이트는 에셋 page-crop 경로에 **이미 존재**(재사용). 요약/뷰어 품질 최선.
  - 단점: PDF 레이아웃(다단·표) 추출 정확도 편차. 추출 실패 시 폴백 필요.
- **C) PDF 우선 + e-print(LaTeX) 폴백.** `arxiv.org/pdf/{id}` → `pdfplumber` 텍스트 추출을 우선, 실패(추출 빈약/PDF 부재) 시 e-print 소스 해제(A안 방식)로 폴백.
  - 장점: 텍스트는 PDF가 더 깨끗 + 폴백으로 가용성 확보. PDF·`pdfplumber`가 에셋 경로에 **이미 존재**(새 의존성 0).
  - 단점: 두 경로 유지 비용, 폴백 시 마크업 잔존 가능. **뷰어 충실도는 여전히 "텍스트 덤프" 수준**(2단 섞임·수식 소실)이라 배포 읽기 품질엔 부족. 스톱갭 성격.
- **D) arXiv HTML 우선 + PDF 폴백(권장).** `arxiv.org/html/{id}`(네이티브) 또는 ar5iv HTML을 우선 취득 → **뷰어는 HTML 렌더**(수식 MathML·표·섹션 보존), **AI 입력은 같은 HTML에서 평문 추출**. HTML 없는 논문(컴파일 실패·구버전)은 **PDF 텍스트 추출(C)로 폴백**.
  - 장점: **뷰어 충실도 최고**(수식·표·구조 보존) + **한 소스로 두 소비자**(뷰어 HTML / AI 평문) 충족 → 중복 취득·불일치 회피. arXiv 소스 이점을 살린 **실제 배포 서비스의 현대적 경로**(arXiv 네이티브 HTML·Semantic Scholar 계열).
  - 단점: **HTML 커버리지 100% 아님** → PDF 폴백 필수(이중 경로). HTML 렌더 시 **XSS 무해화 필수**(외부 콘텐츠 삽입 위협). 평문 파생 단계 추가, 보관물이 형식 2종(HTML+평문) → 작업량 최대.

- **업계 관행(참고)**: PDF만 있는 일반 출판물은 **GROBID**(PDF→구조화)가 표준. arXiv처럼 **소스가 있으면** 더 좋은 패가 있다 — S2ORC는 소스 우선, arXiv 자체도 소스를 LaTeXML로 **HTML화(ar5iv/네이티브 HTML)**. `.tex` 단순 병합(A)은 마크업 잔존으로 오히려 PDF보다 못하다.
- **텍스트·그림 우선순위는 반대**(의도): **그림**은 소스 원본이 고화질이라 *e-print 우선*(현 에셋 경로). **본문**은 *HTML > PDF > LaTeX 원본* 순으로 읽기 좋다.
- **권장**: **D(arXiv HTML 우선 + PDF 폴백)** — 배포 기준에서 뷰어 충실도와 AI 평문을 한 소스로 동시 충족하는 유일한 안. C(PDF 우선)는 뷰어가 텍스트 덤프라 읽기 품질 미달, A/B는 단일 소스라 가용성·충실도 중 하나를 잃는다. **단계적 적용 허용**: HTML 파이프라인 완성 전 #139 깨짐을 즉시 막아야 하면 **C를 스톱갭**으로 먼저 깔고 D로 승격.
- **부수 효과(의식적 수용 필요)**: 보관물이 **HTML(뷰어)+평문(AI)** 2종이 되므로 뷰어 렌더(XSS 무해화)·요약 입력 토큰 추정·#135 청킹이 평문 형식을 전제로 정렬돼야 한다. 비텍스트/추출 실패·HTML 부재 응답은 **검증 단계에서 거르거나 폴백**(현재는 무검증 디코딩).
- **[Answer]**: **D (취득)** — arXiv HTML 우선 + PDF 폴백. #139는 본 결정으로 이관·종료.
  - **적용 범위 확정**: HTML은 *가장 깨끗한 평문을 뽑는 소스*로 사용하고 **보관·소비는 평문 1종**(`.txt`). 뷰어는 평문 렌더(기존 앵커 하이라이트 유지). **리치 HTML 렌더/보관은 보류** — 수식·표·그림 인라인이 **에셋 패널(FR-17)과 그림·표 표시가 겹쳐** 중복 정리 결정이 필요하므로, HTML 리치뷰는 **에이전트 단계**에서 에셋과 함께 설계한다(예방적 보관 회피, 원칙1).

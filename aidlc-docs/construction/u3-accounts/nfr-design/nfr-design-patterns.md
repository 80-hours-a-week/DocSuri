## 1. 세션 저장소(Redis) 연결 복원력 패턴 (Q1 반영)

ElastiCache Redis 연결 지연이나 단절이 시스템 전체의 연쇄 장애(Cascading Failure)로 번지지 않도록 Fail-Closed 및 서킷 브레이커 패턴을 결합합니다.

```
[클라이언트 요청]
       |
       v
[U6 Authn 미들웨어]
       |
       v
[U3 SessionVerifier] ───(서킷 브레이커 열려있음)───> [즉시 거부 (500/401)]
       | (서킷 브레이커 닫혀있음)
       v
[Redis 조회] ───(연결 타임아웃 1.0초 초과 / 에러)───> [에러 카운터 증가] ──> [서킷 브레이커 오픈]
                                                                        |
                                                                        v
                                                                 [즉시 거부 (500/401)]
```

### 1.1. 세션 연결 정책 및 타임아웃
- **소켓 타임아웃**: Redis 클라이언트 연결 및 읽기/쓰기 타임아웃을 **1.0초**로 엄격히 설정합니다.
- **Fail-Closed 강제**: 타임아웃이나 연결 거부 발생 시 RDS 데이터베이스로 조회 폴백을 수행하지 않으며, 해당 요청을 무조건 실패(`SessionStoreException`) 처리하고 HTTP 500 또는 401 오류를 반환합니다.

### 1.2. 서킷 브레이커 (Circuit Breaker)
- **임계치 및 트리거**:
  - 10초 슬라이딩 윈도우 내 연속 **5회** 연결 실패/타임아웃 발생 시 서킷 브레이커가 `OPEN` 상태로 전환됩니다.
  - `OPEN` 상태에서는 Redis에 커넥션을 시도하지 않고 즉시 예외를 던져 API 스레드가 대기 없이 빠르게 응답을 반환할 수 있도록 차단합니다.
  - **하프 오픈(Half-Open) 검증**: 30초 경과 후 단 1개의 요청만 Redis로 전달하여 연결 복구를 확인하고, 성공 시 `CLOSED` 상태로 환원합니다.

## 2. 외부 API 복원력 및 완화 패턴 (Q2 반영)

### 2.1. Google reCAPTCHA v3 (보안 우선, Fail-Closed)
- **타임아웃**: reCAPTCHA 검증 API 호출 타임아웃을 **2.0초**로 디자인합니다.
- **예외 정책**: API 오류, 네트워크 타임아웃 혹은 반환된 봇 점수(Score)가 임계치(예: 0.5) 미만인 경우, 요청을 즉시 중단하고 HTTP 400 또는 429 오류를 리턴합니다. 우회(Soft Bypass)는 허용하지 않습니다.
- **적용 지점(구현 정합, N6)**: reCAPTCHA는 **로그인**의 무차별 대입 게이트(BR-A4 — 10회 연속 실패 시 `X-Recaptcha-Token` 필수)와 계정 복구 경로에 강제합니다. **가입**의 봇/허위 완화는 reCAPTCHA가 아니라 **per-email/per-IP 레이트 리밋 + PENDING 이메일 인증(BR-A5)** 으로 수행합니다(가입에 reCAPTCHA를 추가하려면 별도 슬라이스). 따라서 "가입 및 로그인 모두 reCAPTCHA"는 설계 의도였으나 구현은 위와 같이 분담합니다.

### 2.2. Resend (가용성 우선, 소프트 폴백 및 비동기 복구)
- **타임아웃**: Resend API 호출 타임아웃은 **3.0초**로 디자인합니다.
- **소프트 폴백 프로세스**:
  1. 가입 정보 영속화(RDS `PENDING` 상태)가 완료된 후 Resend 호출을 수행합니다.
  2. Resend 호출이 타임아웃되거나 에러를 반환하는 경우, 가입 트랜잭션을 롤백하지 않고 **성공적으로 가입 완료 응답을 반환**합니다.
  3. 이와 동시에 도메인 레벨에서 `EmailDeliveryFailureSignal`을 발행하여 이벤트 백본(EventBridge/SQS)으로 전송합니다 (SEC-11).
  4. 실패한 전송 요청은 로컬 데이터베이스의 재시도 테이블(`EmailRetryQueue`)에 저장되어, 백그라운드 워커에 의해 최대 3회(지수 백오프 적용: 10분, 30분, 1시간 후) 자동으로 재시도 처리됩니다.
  5. 3회 재시도 실패 시 관리자(OP) 알림을 발생시켜 수동 메일 재발송 처리를 유도합니다.

## 3. 로그인 점진적 시간 지연 패턴 (BR-A4)

로그인 실패 횟수에 따라 CPU 연산 대기를 추가하여 자동화된 무차별 대입 스캔 속도를 늦춥니다.
- **지연 연산 식**:
  - 실패 횟수 `N`이 3회 미만: 즉시 응답 반환.
  - 실패 횟수 `N`이 3회 이상: `asyncio.sleep(2^(N-3))` 초 비차단 대기 강제 (3회=1초, 4회=2초, 5회=4초 …). 상한은 **120초**로 클램프해 무한 증가를 막는다(N1 — 구현 정합: 코드 `min(2^(N-3), 120)`. 과거 문서의 "최대 10초"는 식과 모순이라 정정). 10회 이상은 §2.1 reCAPTCHA 게이트가 추가된다.
  - 지연 로직은 데이터베이스 커넥션을 점유하지 않도록 트랜잭션이 **완전히 종료(Commit)된 이후** 컨트롤러 레이어 반환 직전에 실행되어야 합니다.

## 4. 비밀번호 재설정 보안 패턴 (BR-A8 / FR-26) *(2026-06-24)*
- 토큰은 평문 비교가 아닌 **해시 저장·해시 조회**(DB 유출 시 토큰 무력화). **단일 사용** = `used_at` 세팅을 트랜잭션 내 원자적 compare-and-set으로 처리(동시 사용 경합 방지). 만료(30분)·used 토큰은 거부.
- 요청·확정 모두 **계정 열거 방지**(가입/상태 무관 동일 응답), 요청에 **레이트 리밋**(IP/이메일 버킷, SEC-11). 성공 시 `userId`의 전 세션 삭제(SessionStore).
- **Fail-Closed**: 메일 발송 실패 시에도 요청 응답은 일반 성공 형태 유지(열거 방지), 발송 실패는 비동기 로깅(SES/Resend 소프트 폴백 패턴 §2.2 계승).

## 5. 소셜 로그인(OIDC) CSRF·신원 검증 패턴 (BR-A9 / FR-27) *(2026-06-24)*
- `state`(CSRF)·`nonce`(replay)는 **서버측 단명 저장**(Redis TTL, 콜백 1회 소비). 콜백 시 `state` 불일치·재사용은 거부.
- `id_token`은 **JWKS 공개키 서명 검증** + `aud`(client_id)·`iss`(Google)·`exp`·`nonce` 검증. JWKS는 **TTL 캐시**로 외부 호출 최소화.
- `email_verified=false` → 자동 연결 거부(계정 탈취 방어). 신규 계정은 `ACTIVE`·역할 `USER` 고정.
- 프로바이더 장애/타임아웃 → 명시적 에러·로그인 화면 복귀(**Fail-Closed**, 부분 계정 상태 미생성).

## 6. 계정 삭제 이벤트 캐스케이드·유예 파기 패턴 (BR-A11 / FR-28) *(2026-06-24, H2 정정 2026-06-30)*
- 삭제 = 2상: ① **동기** 소프트 삭제(`DEACTIVATED`·전 세션 무효화·유예 레코드 생성) ② **비동기** 유예 파기 잡. **`AccountDeleted`는 ①이 아니라 ②(영구 파기 시점)에 발행한다 (H2 — 유예 동안 owner-scoped 데이터를 보존해 복구 가능하게).** (이 절의 과거 표현 "① …·`AccountDeleted` 발행"은 H2 리뷰 결정 이전의 잔재이며, 코드·`shared/events` 계약은 ②에서 발행한다.)
- 캐스케이드는 **이벤트 구동**(EventBridge): 구독자(U4/U2/U11/U12)는 **멱등**(이미 파기 시 무시, `eventId`=uuid5(account_id))·재시도·**DLQ**. U3는 구독자 완료를 대기하지 않는다(비차단·결과적 일관성). 같은 RDS의 owner-scoped 테이블은 `SqlOwnerDataPurger`가 파기 트랜잭션에서 동기 backstop으로 정리한다.
- 유예 파기 잡: `purge_after` 경과분을 행별 트랜잭션으로 처리(멱등). 발행/파기 반복 실패는 시도 횟수를 누적해 임계 초과 시 `PURGE_FAILED`(DLQ)로 격리한다(S2).

## 7. 공개 인증 입력 견고화 패턴 (BR-A12 / FR-29 — 현 로그인 422 장애 해소) *(2026-06-24)*
- 공개 인증 입력 DTO는 **`extra=ignore`**(SSOT 스키마 `additionalProperties: true` 재생성·§5-B) → 추가 필드 무시·필수 필드만 강제. **인가·소유권 경로 DTO는 `extra=forbid` 유지**(완화 안 함 — 파라미터 스머글링 방어 한정 적용).
- 컨트롤러는 422/4xx를 안정적 에러 코드 + 비기술 메시지로 매핑; 프런트 `normalizeHttpError`가 **422를 명확 메시지로 표면화**(현 'unknown' dead-end 해소) + 인증 메일 재발송 경로.

## 1. 성능 요구사항 (Performance Requirements)

### 1.1. 세션 검증 레이턴시 예산 (NFR-P1)
모든 API 호출 시 미들웨어(U6)를 통해 사용자 인증 유무 및 토큰 유효성을 판정하는 `SessionVerifier.verifyRequest`는 서비스 전체의 병목을 방지하기 위해 극도로 높은 성능을 만족해야 합니다 (Q1 답변 반영).
- **지표**:
  - **P50 Latency**: **5ms 미만**
  - **P99 Latency**: **20ms 미만**
- **제약**: 세션 데이터 검증 시 디스크 I/O를 발생시키지 않고 인메모리 스토리지에서 즉시 조회 및 갱신(Sliding Expiration)을 완료해야 합니다.

### 1.2. 자격증명 해싱 및 API 응답 레이턴시
로그인 및 회원가입은 암호학적 연산(Argon2id KDF)으로 인해 고의적인 CPU/메모리 부하가 수반되며, 이는 성능과 보안 간의 의도적인 트레이드오프입니다.
- **Argon2id 연산 시간**: 사용자당 단일 코어 기준 **100ms ~ 200ms** 사이로 실행 속도를 튜닝하여, GPU를 이용한 대규모 패스워드 무차별 대입 공격을 차단합니다.
- **Login/Signup API 응답 속도**:
  - **P50 Latency**: **300ms 미만** (KDF 실행 속도 포함)
  - **P99 Latency**: **1000ms 미만** (네트워크 지연 포함)

## 2. 확장성 및 부하 요구사항 (Scalability Requirements)

- **동시성 세션 용량**: 시스템은 최대 **10,000개**의 동시 활성 사용자 세션을 유성적으로 보관 및 처리할 수 있어야 합니다.
- **처리량 (Throughput)**: 세션 검증 API는 최소 **100 RPS (Requests Per Second)** 이상의 유효성 검증 요청을 지연 없이 처리할 수 있어야 합니다.
- **계정 규모**: 1차 프로덕션 타깃 기준 최대 1,000명의 활성 사용자를 지원하며, 스토리지 용량은 인덱스 및 자격증명 메타데이터를 포함해 사용자당 10KB 미만으로 엄격히 제한합니다.

## 3. 가용성 및 복원력 요구사항 (Availability & Resilience)

### 3.1. 세션 데이터 가용성 (Q3 반영)
세션 정보가 유실될 경우 현재 활성 사용자가 강제 로그아웃되는 대규모 사용자 경험 저하가 발생하므로, 인메모리 스토리지 복제 및 다중화를 통해 세션 상태의 가용성을 극대화합니다.
- **가용성 목표**: **99.99%**
- **장애 복구 목표**:
  - **RTO (Recovery Time Objective)**: 1분 미만 (장애 조치 / Auto-Failover 자동화 적용)
  - **RPO (Recovery Point Objective)**: 0 (활성 세션의 실시간 Multi-AZ 복제를 통해 유실 제로 보장)

### 3.2. 자격증명 데이터 가용성
사용자 계정 및 해시가 저장된 주 데이터베이스는 RDBMS의 고가용성 멀티 AZ 가용 영역 구성을 준수합니다.
- **가용성 목표**: **99.99%**
- **백업 및 복구 목표**:
  - **RPO**: 24시간 이내 (매일 스냅샷 및 증분 백업)
  - **RTO**: 4시간 이내 (재해 복구 시나리오 준수)

## 4. 보안 및 위협 모델링 요구사항 (Security Requirements)

- **전송 보안**: 모든 계정 관련 API 호출 및 토큰 전송은 TLS 1.2+ (HTTPS) 상에서만 수행되어야 합니다 (SEC-1).
- **세션 쿠키 정책**: 브라우저에 저장되는 세션 토큰 쿠키는 반드시 `HttpOnly`, `Secure`, `SameSite=Lax` 속성을 설정하여 XSS 및 CSRF 공격을 방어합니다 (SEC-12).
- **무차별 대입 완화**:
  - 3회 실패 시부터 `2^(실패 횟수 - 3)` 초 지연 적용 (최대 10초 대기).
  - 10회 실패 시 Google reCAPTCHA v3 토큰 검증 필수 적용 (Q5 반영).
- **로그 정제 정책 (SEC-3)**:
  - 평문 비밀번호 및 세션 토큰은 감사 로그(audit.md) 및 구조화 로그에서 정규식 및 이음새를 통해 완전히 차단/필터링되어야 합니다.
- **예외 페일세이프 (SEC-15)**:
  - RDS 또는 Redis 스토리지 장애 등으로 세션 데이터 획득에 실패할 경우, 시스템은 항상 사용자 권한을 거부(`DENY`, 401 Unauthorized)하고 예외를 전역으로 캐치하여 비기술적인 안전한 오류 코드를 응답합니다.

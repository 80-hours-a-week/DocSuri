## CQ4 — RTO/RPO 목표 및 재해 복구 전략  _(RESILIENCY-02 — 필수)_
RTO(복구 시간 목표)와 RPO(복구 시점 목표) 목표는? 이는 적절한 재해 복구 전략과 인프라 이중화 수준을 결정한다.

_중간 티어 공개 디스커버리 도구로서, 지속 보존되는 사용자 데이터가 계정 + 검색 저장뿐이라면 권장안: **E**(단일 리전, 멀티 AZ) + 자동 일일 DB 백업 — AZ 장애를 견디는 최저 비용 옵션. 시간 단위 RTO의 명시적 백업·복원 DR을 원하면 A._

A) RPO/RTO: 시간 단위 — Backup & Restore 전략. 최저 비용($). 데이터 백업, 서비스 미배포. 장애 시 IaC로 재배포 + 백업 복원. 비핵심 워크로드 적합.

B) RPO/RTO: 수십 분 — Pilot Light 전략. 비용 $$. 데이터 라이브, 서비스 유휴. 인프라 배포되나 미가동, 페일오버 시 스케일업. 중요 워크로드 적합.

C) RPO/RTO: 분 단위 — Warm Standby 전략. 비용 $$$. 데이터 라이브, 서비스 축소 가동. 페일오버 시 스케일업. 비즈니스 핵심 애플리케이션 적합.

D) RPO/RTO: 근실시간 — Multi-site Active/Active 전략. 최고 비용($$$$). 데이터 라이브, 여러 리전 동시 라이브 서비스. 미션 크리티컬·무중단 요구 적합.

E) N/A — 단일 리전 배포 수용 가능, 교차 리전 DR 불필요. 한 리전 내 멀티 AZ 가용성에 의존.

X) 기타 (아래 [Answer]: 태그 뒤에 기술)

[Answer]: E

## CQ5 — 변경 관리 프로세스  _(RESILIENCY-03 — 필수)_
이 워크로드의 프로덕션 변경을 어떻게 통제할까? AI-DLC는 새 프로세스를 만들기보다 답변에 설계를 맞춘다.

_권장: **A**, 기존 **GitHub PR 리뷰 + git-flow**(feature → develop → main)를 변경 프로세스로 명명 — 이미 사용 중._

A) 기존 조직 변경 관리 프로세스 사용 — 이름/도구 제공(예: ServiceNow, Jira Change, 내부 CAB). AI-DLC가 이를 참조하고 배포 산출물이 그 프로세스(변경 기록, 승인 게이트)에 맞도록 보장.

B) 아직 정식 프로세스 없음 — AI-DLC가 경량 변경 관리 프로세스(변경 기록 + 승인 + 롤백 노트)를 제안.

C) N/A — 이 워크로드는 정식 변경 관리 면제(예: 내부 도구). 면제 사유 문서화.

X) 기타 (아래 [Answer]: 태그 뒤에 기술)

[Answer]: A. GitHub PR review + git-flow + GitHub Projects.

_권장: **B** — 팀에 아직 정식 IR/온콜 도구가 없으므로 AI-DLC가 경량 장애 대응 + 오류 교정(COE) 프로세스를 제안._

B) 정식 프로세스 없음 — AI-DLC가 경량 장애 대응 및 오류 교정(COE) 프로세스를 제안.

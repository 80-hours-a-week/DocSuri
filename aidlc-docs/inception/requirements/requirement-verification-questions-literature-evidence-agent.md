**담당**: 화랑님
**대상 기능**: 로그인 필수 온디맨드 **대화형 다논문 문헌탐색·근거형성** — 사용자 질의/첨부에 대해 여러 논문을 검색·교차확인하여 **추출 기반 근거**(핵심 주장·방법·결과 수치·한계)를 **비교 정리**해 출처와 함께 제시하고, 근거가 없으면 **기권**한다. 결과·대화 세션을 owner-scoped로 영속(전용 네비 진입). UI = **채팅 형식**(모드 선택).

**근거 SSOT**: 재인셉션 차터 `inception/plans/reinception-2026-06-charter.md`(페이즈 4·§4·D2/D5) · 코드 베이스라인 `inception/reverse-engineering/code-baseline-2026-06.md`(§3 계약 인벤토리) · 공유 계약 골격 `inception/application-design/agent-tool-port-contract-draft.md`.

## A. 산출물 · 공유 계약 (★5가 잠그는 핵심★)

### Q1. 근거형성 약속 수준 — 무엇을 끌어내나
다논문에서 추출·정리할 근거의 폭을 어디까지로 하는가?

- **A) 핵심 주장 · 방법 · 결과 수치 · 한계** (초록에 안 나오는 본문 근거 중심). **재현성은 판정/계산 제외** — 선택적 "코드/데이터 공개 *사실* 추출"만. *근거: 합성 판정은 FR-5 그라운딩에 부적합.* (권장)
- **B) A + 재현성 정성 판정** (재현성 점수/판정 포함). *트레이드오프: 가치↑이나 합성 판정 환각 위험↑, FR-5 정신과 긴장.*
- **C) 최소 — 핵심 주장 · 결과만** (방법·한계 생략). *트레이드오프: 단순·저비용이나 "초록+α" 수준에 그쳐 차별성↓.*
- **X) 기타**

[Answer]: **A** — 핵심 주장·방법·결과 수치·한계 추출. 재현성 판정은 합성 판단이라 FR-5 그라운딩 원칙(추출 기반·기권 강제)과 충돌하므로 제외. "코드/데이터 공개 사실" 추출은 허용(판정 아님).

### Q2. 근거표(비교 정리) 컬럼 스키마
근거를 어떤 형태로 비교 정리하는가?

- **A) 논문 비교형**(행=논문, 열=주장/방법/결과/한계) **+ 쟁점 오버레이**(논문 간 일치/상충 표시). *표현력 최대, 구현·렌더 복잡도 중.* (권장)
- **B) 쟁점 매트릭스**(행=쟁점, 열=논문별 입장). *쟁점 중심엔 강하나 논문 단위 조망↓.*
- **C) 최소 리스트**(논문별 근거 항목, 표 아님). *가장 단순, 교차 비교 가치↓.*
- **X) 기타** _↪ 정확한 컬럼/깊이는 Functional Design._

[Answer]: **A** — 논문 비교형(행=논문, 열=주장/방법/결과/한계) + 쟁점 오버레이(일치/상충 표시). 논문 단위 조망과 교차 비교를 동시에 제공하는 것이 문헌탐색 에이전트의 핵심 가치. 컬럼 상세·렌더 방식은 Functional Design에서 확정.

### Q3. 근거 출력 DTO 깊이 (★연구아이디어 유닛에 노출★)
5가 소비할 `EvidenceItem`을 어디까지 담는가? *(`SourceRef`=IndexRecord+DocModel Block id 재사용은 고정 — 골격 §3.)*

- **A) 풍부**: `EvidenceItem{ statement, supporting[], conflicting[], confidence? }` — 교차확인(상충)·신뢰도 표현 최대. *5의 novelty 판단 입력 풍부 / 스키마·환각 관리 부담.* (권장)
- **B) 중간**: `{ statement, supporting[], conflicting[] }` (confidence 제외). *상충은 표현하되 정량 신뢰도 환각 회피.*
- **C) 최소**: `{ statement, supporting[] }` (conflicting·confidence 제외). *단순하나 교차확인 표현 불가 → 5 가치↓.*
- **X) 기타** _↪ confidence 정량화·상충 표현 깊이는 Functional Design; 생성 산문 금지(C-2) 유지._

[Answer]: **B** — `{ statement, supporting[], conflicting[] }` (confidence 제외). 상충 표현(conflicting[])은 novelty 판단에 필수라 포함. confidence는 LLM이 합성하는 값으로 환각 위험이 높고 FR-5 그라운딩 원칙과 긴장하므로 제외. confidence가 필요하면 5 유닛이 자체 로직으로 산출하도록 위임.

## B. 입력 · 검색

### Q4. 검색 scope (근거 모을 논문 집합)
근거형성 대상 논문을 어떻게 모으는가?

- **A) 혼합** — 기본 검색 주도 자동(질의→다논문) + 사용자 명시 paper 집합/첨부 허용. *유연·UX 좋음 / 두 경로 처리.* (권장)
- **B) 검색 주도 자동만**. *단순하나 "이 논문들로" 요구 대응 불가.*
- **C) 명시 paper 집합만**. *통제적이나 탐색 가치↓(에이전트보다 뷰어에 가까움).*
- **X) 기타**

[Answer]: **A** — 혼합. 질의 주도 자동 검색을 기본으로 하되, 사용자가 "이 논문들로" 지정하거나 PDF를 첨부하면 해당 집합도 근거 대상에 포함. 두 경로는 동일한 `EvidenceItem` 계약으로 통합 출력. 사용자 명시 집합 없이 자동 검색만 진행하는 것도 유효한 흐름(Q6 C 제외 시).

### Q4b. 자연어 질의 이해(NL 질의 → 검색 의도)의 소재 — **결정됨 (2026-06-29)**
"reinforcement 초보자용 논문 추천해줘" 같은 **대화형 자연어 질의**의 의도 해석(군더더기 제거 · "초보자용 = survey/tutorial" 해석 · 검색어 정련)을 **어디서** 수행하는가?

- **A) 에이전트 내부 신규 단계** — LLM 질의 이해 → 정련된 검색어로 discovery Search Tool을 `scope=FULL` 호출. U2 discovery(full 포함)는 **무수정**. *경계 명확 · U2 동기 P50<3s 예산 보존 · 사람 검색창과 분리.* (결정)
- **B) U2 `scope=FULL` 경로에 LLM 질의 이해/재작성 추가**. *U2 동기 P50<3s 예산 위반 · U2 Q1=A(LLM 재작성 없음) 결정과 충돌 · 검색 엔진에 대화 해석 책임 혼입 → 기각.*

[Answer]: **A** — 질의 이해는 **에이전트 책임(신규 구현)**, U2는 기존 `POST /api/search`를 `scope=FULL`로 **Tool 소비만**(full 포함 코드 변경 없음).
근거:
- `scope=FULL`은 **에이전트가 쓰도록 U2에 선제 구현된 검색 모드**다(본문까지 고recall 검색). 단 "에이전트용으로 만들었다 ≠ 에이전트 코드에 둔다" — 벡터 ANN·BM25·RRF·인덱스 연결은 U2 소유이므로, full도 **U2 엔진의 한 모드**로 살고 에이전트는 **호출만** 한다(provider/consumer). 에이전트에 full을 재구현하면 검색 엔진 전체를 복제하게 된다.
- LLM 질의 이해는 검색 *앞단 전처리*이지 검색 *깊이(scope)*와 무관 — full에 넣을 성질이 아니다. U2 Q1=A(LLM 질의 재작성 없음)·Q3=A(LLM 리랭킹 없음) 그대로 유지. _↪ Tool 호출이 HTTP냐 in-proc냐는 §공유 계약/Functional Design._

### Q5. 코퍼스 커버리지 범위 (v1)
근거 소스 범위를 무엇으로 하는가?

- **X) 기타**

[Answer]: **A** — owned arXiv AI/ML 전문 인덱스만(페이즈 1 Corpus·DocModel 재사용). v1은 외부 API 없이 기존 인프라로 근거 충실도와 저비용을 동시에 확보. 전 분야 커버리지 확장은 페이즈 5(연구아이디어 유닛) 사안으로 이월.

### Q6. 문서 첨부 처리
첨부(사용자 초안/논문)를 어떻게 다루는가?

- **A) doc-model 파이프라인 재사용 파싱** + 형식/크기 검증·무해화 + owner-scoped 격리. *기존 자산 재사용·구조 보존.* (권장)
- **B) 텍스트만 단순 추출**(doc-model 미사용). *가벼우나 표·수식 근거 손실.*
- **C) v1 첨부 비지원**(질의만). *범위 축소 — 단 "내 초안 기준" 가치 상실.*
- **X) 기타** _↪ 형식/크기 한도·파싱 폴백은 Functional Design._

[Answer]: **A** — doc-model 파이프라인 재사용. 기존 자산으로 표·수식 포함 구조를 보존하고, 형식/크기 검증·무해화·owner-scoped 격리는 SEC-5/11 상속 제약(C-6)으로 처리. 형식별 크기 한도와 파싱 폴백은 Functional Design으로 이월.

### Q7. 멀티턴 맥락
대화 세션 안에서 이전 턴을 어떻게 잇는가?

- **A) 세션 내 멀티턴**(이전 턴 근거/대상 논문 맥락 누적, 후속 질문). *대화 가치↑ / 맥락 관리·비용.* (권장)
- **B) 단발 Q→근거**(턴 간 맥락 없음). *단순·저비용이나 "대화형" 약속과 약함.*
- **X) 기타** _↪ 맥락 윈도/요약 전략은 NFR Design._

[Answer]: **A** — 세션 내 멀티턴. 이전 턴의 근거·대상 논문 맥락을 누적해 후속 질문("그 논문 중 한계가 가장 적은 건?")에 대응. "대화형" 약속의 핵심 가치이므로 B(단발)는 불가. 맥락 윈도 전략·토큰 예산 관리는 NFR Design으로 이월.

### Q9. 동기/비동기 응답 표면 (온디맨드·비차단 — 검색 SLA 비대상)

## D. 영속 · 보관

### Q10. 보관 · 삭제 정책
첨부 원본·대화·결과 보관을 어떻게 하는가? *(owner-scoped 영속·전용 네비 재열람 자체는 상속 — 아래 확인 블록.)*

- **A) 보관 + 사용자 삭제/초기화 제어 필수**(기간 정책은 운영 이월). *재열람 가치↑ / 저장·프라이버시 관리.* (권장)
- **B) 세션 종료 시 첨부 폐기**(결과만 보관). *프라이버시↑·저장↓ / 첨부 기준 재분석 불가.*
- **C) 기간 제한 자동 만료**(예: N일 후 파기). *저장 통제 / 만료 정책·UX 필요.*
- **X) 기타** _↪ 보관 기간 수치·파기 잡은 NFR/Infra Design; 계정삭제 캐스케이드는 U3 `AccountDeleted` 구독으로 편입._

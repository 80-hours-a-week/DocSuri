**상태**: 🟡 DRAFT 제안 · **범위**: 신규 유닛(U7) — 현 frozen 범위 밖, 요구사항 등재 필요 · **일자**: 2026-06-18

> **🔄 개정 (2026-06-19) — 제품 방향 변경 (본 개정이 §2~§12의 상충 항목에 우선한다)**
> 오너 결정으로 아래가 확정·변경됨. 이 블록이 SSOT의 최신 진실이며, 아래 본문의 상충 기술(persona·번역 범위)은 본 개정으로 대체된다.
> 1. **번역 = 초록 + 전문 둘 다.** 기존 "번역=초록만"을 확장 — `translate` task에 `scope: "abstract" | "full"`(기본 `abstract`) 추가. 요약=전문 대상은 그대로.
> 2. **persona(전문가/입문자) = 요약에만 유지, 번역은 단일.** 요약은 수준 2벌(전문/입문) 생성, 번역(초록·전문)은 수준 분기 없음 — 번역은 충실성 작업이라 persona 부적합(입문자 친화는 용어집으로). §9.2 "생성 변형(persona 2벌)"은 **요약 한정**으로 유지. (초기 '전면 제거' 결정을 요약 한정으로 재수정.)
> 3. **화면 구조 확정(프론트 FD 입력).** 카드 = `[요약]` 탭 → tldr(3줄) 인라인 + `상세히 보기` 링크. 상세 페이지 = 전체 구조화 요약 + 출처보기(인라인 앵커) + [초록 번역] + [전문 번역].
> 4. **출처보기 = 전문 하이라이트 뷰어(Q5=C 확정).** 앵커 클릭 → 원문 전문에서 위치 하이라이트. ⚠️ 신규 백엔드 의존: **전문 텍스트 반환 API**(현재 S3 private·미반환) + **OA 라이선스 게이트** + 정규화 텍스트 가독성 처리. (초기 '인라인 앵커·뷰어 없음'에서 전환.)
> 5. **캐시 키**: `(paperId, version, task, lang, scope, persona, glossaryVer, modelVer, promptVer)` — 요약: persona∈{expert,beginner}·scope=full / 번역: persona=none·scope∈{abstract,full}. 각 조합 = 별개 영구 객체.
> 6. **비용**: 전문 번역 라인 신설(§8) — Haiku, ≈$0.1(보통)~$0.3(장문)/편, 온디맨드+영구저장으로 bounded.
**전제**: 검색 인덱스는 **(A) 초록 임베딩(논문당 1벡터)** 으로 단순화한 뒤의 설계.
**SSOT 근거**: `aidlc-docs/inception/.../unit-of-work.md`(유닛 U1~U6) · `aidlc-docs/construction/shared/ports.md`(U6 근거화·비용·관측 단일 권위) · `aidlc-docs/inception/requirements/requirements.md`(FR-5·C-2·NFR-C1·SEC-11) · `vector-spec.md`(전문 원본 S3 보관)

> ⚠️ **요약/번역은 벡터 RAG가 아니다.** 대상 논문을 이미 알고 있으므로(사용자가 결과 카드를 탭) "검색"이 필요 없다.
> **"S3의 그 논문 전문 1편 → LLM → 한국어 요약"** 의 단일 문서 처리다. 따라서 (A)로 청크 임베딩을 안 해도 이 기능은 그대로 성립한다.

> 🎯 **설계 프레임**: "논문을 요약한다"가 아니라 **"페르소나(P1 박지훈)의 4가지 질문에 답한다"** —
> ① 내 아이디어와 겹치나(novelty) ② 정확히 뭘 했나(방법) ③ 실제로 되나(결과) ④ 위에 쌓을 수 있나(재현성·한계).
> 초록은 이미 일반 요약 역할을 하므로, 본 기능의 가치는 **초록에 안 나오는 디테일(결과 수치·한계·재현성)을 끌어내는 것**이다.

> 📐 **방법론 주의**: 요약/번역은 검색처럼 **점수(recall@k 등)로 측정→튜닝**하며 아키텍처를 찾는 기능이 *아니다* → **설계로 확정**하는 기능이다. 따라서 본 문서는 "일단 minimal, 추후 개선"이 아니라 **완결 스펙**을 목표로 한다. 측정이 필요 없는 결정은 §12에서 모두 **확정**으로 적는다.

## 🔎 한눈에 보기 — 흐름 (먼저 이것만 보면 됨)

**등장인물 4개**: 🗄️Redis(자주 본 요약 임시보관) · 📦S3(논문 전문 + 완성 요약 영구보관) · 🤖Haiku/🧠Sonnet(번역=Haiku·요약=Sonnet, §2) · 🚦비용 게이트(돈 초과 확인 신호등)

```
 👤 카드 [요약] 탭 → tldr 3줄 인라인 + [상세히 보기]
      상세 페이지의 3개 액션(모두 아래 같은 흐름):
      ├─ [요약]      → 논문 전체를 구조화 요약 (수준: 전문가용 / 입문자용)
      ├─ [초록 번역]  → 초록을 한국어로            (scope=abstract)
      └─ [전문 번역]  → 논문 전문을 한국어로        (scope=full · 신규)
            │
            ▼
 ┌───────────────────────────────┐
 │ ①  이미 만들어둔 결과 있나?     │   🗄️Redis → 없으면 📦S3
 └───────────────────────────────┘
      ├─── 있음 ───────────────────────▶  📲 바로 표시   ✅ (공짜)
      ▼ 없음
 ┌───────────────────────────────┐
 │ ②  지금 AI 돌려도 되나?         │   🚦비용 게이트
 └───────────────────────────────┘
      ├─── 안 됨(비용초과) ─────────────▶  📲 "잠시 후 다시"   ⛔
      ▼ 됨
 ┌───────────────────────────────┐
 │ ③  원문 가져오기                │   요약=📦S3 전문 / 번역=초록
 └───────────────────────────────┘
      ▼
 ┌───────────────────────────────┐
 │ ④  본문 청소                    │   참고문헌·깨진 수식 제거, 표/그림 캡션은 남김
 └───────────────────────────────┘
      ▼
 ┌───────────────────────────────┐
 │ ⑤  AI 처리                      │   요약 → 🧠Sonnet : 기여·결과·한계·재현성 구조로
 │                                 │   번역 → 🤖Haiku + 📖용어집
 │                                 │           (Transformer 등 전문용어는 영어 그대로)
 └───────────────────────────────┘    ※ AI 장애·지연 → 타임아웃 → "잠시 후 다시"(저하)
      ▼
 ┌───────────────────────────────┐
 │ ⑤-1  근거 OK?                   │   주장이 원문에 실재? (날조 검사)
 └───────────────────────────────┘
      ├─ OK ──── 스트리밍 ────▶  📲 결과 + 항목별 "출처 보기"(원문 하이라이트)
      │                            └ 보기 전환(재생성 0): 전체 / 3줄 / 성능·구현·리뷰어 관점
      └─ 실패 ─────────────────▶  📲 "근거 부족" 기권(날조 방지)   ⛔
      ▼ (OK)
 ┌───────────────────────────────┐
 │ ⑥  완성본 영구 저장             │   📦S3 + 🗄️Redis → 다음 사람 공짜
 └───────────────────────────────┘
```

**핵심 4줄**
- **①에서 대부분 끝난다** — 이미 만든 게 있으면 AI 안 돌리고 즉시 표시(공짜).
- **AI가 실제로 도는 건 ⑤ 한 곳** — 그 논문을 *처음* 처리하는 사람일 때만(이후 영구저장으로 공짜).
- **요약=구조화**(기여/결과/한계/재현성), **번역=용어집으로 전문용어 일관**(미번역 단어 영어 유지). 결과엔 항목별 **"출처 보기"**로 원문 검증 가능.
- **모델은 작업으로 자동**(요약 🧠Sonnet / 번역 🤖Haiku) — 사용자가 안 고름. ⑤-1은 **날조 검사**(실패 시 기권).
- **개인화**: 요약 *수준*(전문가용/입문자용)은 선택 → 그 버전 생성(논문당 ≤2벌). **번역은 단일**. *보기*(3줄·관점별)는 같은 결과를 다르게 표시할 뿐 **재생성 0**.

> 아래 §1~§12은 위 흐름의 **상세·근거**다. 전체 그림이 잡혔으면 필요한 섹션만 펼쳐 보면 된다.

## 2. LLM 모델 선택

> ⚠️ **코퍼스 전제**: 대상은 **AI/ML·통계·데이터사이언스·CS**의 arXiv 논문(C-6 슬라이스: cs.LG/cs.AI/cs.CL/cs.CV/stat.ML). **수식·이론·새 방법이 밀집한 난해 코퍼스**다. "대부분 쉬워서 Haiku로 충분"은 **틀린 가정** — 특히 *요약*은 미묘한 기여·한계를 정확히 잡아야 해 모델 역량이 중요하다.

**모델은 작업으로 정해진다**(번역=Haiku / 요약=Sonnet). 사용자에게 선택기를 노출하지 않는다(대부분 이해 못 하고 캐시도 쪼개짐).

| 작업 | 모델 | 근거 |
|---|---|---|
| **번역** `[한국어로]` | **Haiku 4.5** (`claude-haiku-4-5`) | 추론 부담 적음 — 품질은 모델보다 **용어집(§9.1)**이 좌우 |
| **요약** `[AI 요약]` | **Sonnet 4.6** (`claude-sonnet-4-6`) | **난해 코퍼스 → 정밀모델 필요**. 연구자 대상이라 품질이 핵심 |

- **품질 게이트(§7) = "근거화 강제"이지 "모델 승격"이 아니다**: Sonnet 요약도 날조 가능 → **모델 무관, 매 출력 검사**(FR-5/QT-1). 실패 → **1회 재시도 → 기권**(fail-closed, NFR-R1). Sonnet 위로 올릴 모델이 없으므로 요약엔 "승격" 개념 없음 — **통과 아니면 기권**.
- **비용 bounded**: 온디맨드 + 영구저장(§11) → "열어본 distinct 논문 × 1회"만 과금 → 요약 Sonnet도 감당.
- **레이턴시**: 요약은 NFR-P1(검색 SLA: P50<3s) **대상 아님**(온디맨드 액션). 체감은 **스트리밍(빠른 TTFB)+캐시 즉시응답**으로 관리. 첫 생성(Sonnet 풀논문)은 수십 초 가능 → **스트리밍 필수**, 초장문(map-reduce)만 비동기 잡(§12 확정).
- **"길이"는 모델 문제 아님**: 둘 다 롱컨텍스트 → 논문 1편(~13K, 길어도 ~40K토큰) 통째 처리. 모델 축은 *난이도*지 길이 아님. 초장문만 map-reduce.
- **AWS 정합**: Bedrock 경유 → IAM·관측·비용 일원화.

## 3. 출력 스키마 — 페르소나 질문 기반

제네릭 요약(❌)이 아니라 P1이 실제로 던지는 질문에 답하는 구조화 출력. 각 항목엔 **근거 앵커**(섹션·표·그림)를 달아 "출처 보기"를 지원(§7).

```json
{
  "tldr":          "문제 + 핵심 주장 (1~2문장)",
  "contributions": ["저자가 주장하는 기여 (이 논문 자체)"],
  "method":        "접근법 한 단락",
  "results":       "핵심 수치·벤치마크·SOTA 여부",   // 초록에 잘 안 나오는 디테일
  "limitations":   "한계 & 가정 (재현성·일반화 경계)", // 페르소나 최우선
  "reproducibility": { "code": "공개?/링크", "data": "공개?/링크" }, // 킬러 포인트
  "anchors":       [{ "field": "results", "target": "section 5.2 | table 3 | figure 2", "span": "…" }]
}
```

> **핵심 차별점**: 초록이 이미 일반 요약이므로, 프롬프트는 **결과 수치·한계·재현성처럼 초록에 없는 정보를 본문·표에서 끌어내도록** 명시 지시한다.
>
> 이 풍부한 구조화 출력 한 벌이 **표시 프리셋(3줄·관점별)의 원천**이다 — 클라이언트가 필드를 골라 보여줄 뿐 재생성 없음(§9.2).

## 5. 사용자 액션 & 무엇을 저장하나

상세 페이지 액션 **3개**(카드는 [요약] 진입 + tldr 인라인, [상세히 보기]로 상세).

| 버튼 | 하는 일 | 입력 | 모델 | **우리가 저장?** |
|---|---|---|---|---|
| **[요약]** (persona: 전문/입문) | 논문 전체를 구조화 요약(§3) | S3 전문 | **Sonnet**(난해 코퍼스) | ✅ **요약 결과** 저장(수준별 ≤2벌) |
| **[초록 번역]** (`scope=abstract`) | 초록을 한국어로 | 초록(이미 보유) | Haiku | ✅ **초록번역** 저장 |
| **[전문 번역]** (`scope=full`) | 논문 전문을 한국어로 | S3 전문 | Haiku | ✅ **전문번역** 저장(별개 키) |

**무엇을 저장하고 무엇을 안 하나 (명확화)**
- ✅ 저장: **우리가 LLM으로 생성한 산출물** = ① 번역 결과, ② 요약 결과. (S3 영구 + Redis 핫, §11)
- ❌ 안 함: **초록 원문** — 이미 검색 인덱스/원문(S3)에 존재. 따로 복제 안 함.

## 7. 품질 판정 & 근거 앵커 (할루시네이션 방어 = 신뢰 UX)

### 7.1 "품질 미달"을 어떻게 판단하나 (계단식)

완벽한 자동 판정은 불가 — **싼 것부터 계단식**으로, 재시도/기권 트리거는 1차로.

**1차 — 결정적 체크 (공짜·코드, 매 건)**
- **앵커 실재성** ⭐: 각 주장이 가리킨 원문 **섹션·표·그림**이 **실제로 존재하나?** 없으면 = 날조 → 실패. (FR-5/QT-1 강제와 일석이조 — 공짜로 가짜 인용 차단)
- **수치 일치** ⭐: 요약의 핵심 수치(벤치마크·metric)가 원문(본문·표)에 **존재하나?** 정규화 비교(95.3% ↔ 0.953 허용). 없으면 = 날조 → 실패.
- **스키마 완전성**: JSON 계약(기여/결과/한계/재현성) 필수 필드 누락 → 실패.
- **잘림·빈 출력·비정상 길이** → 실패.

**게이트는 결정적 체크(앵커+스키마)만.** LLM-judge(NLI 등)는 쓰지 않는다 — 전건 적용 시 콜 2배.

**판정 → 행동** (게이트 목적 = **근거화 강제**, 모델 무관·필수)
- 1차 통과 → 저장.
- **정당한 기권**(코퍼스 밖이라 abstain) = 정상 동작, 재시도 안 함.
- **1차 실패** → **1회 재시도 → 그래도 실패 시 기권**(fail-closed). 근거 없는 출력은 사용자에게 안 보냄.

> **한계(정직)**: 1차는 *구조 깨짐·가짜 인용*은 잘 잡지만, **그럴듯한데 미묘하게 틀린 요약**(방법 왜곡)은 못 잡는다. 이 빈틈은 자동 완전해결 불가 → **앵커("출처 보기")로 사용자가 직접 검증**하게 두어 보완.

### 7.2 근거 앵커 = 신뢰 UX

- 각 출력 항목(`anchors[]`)이 원문 **섹션/span**을 가리킴 → 클라이언트 "출처 보기" → 원문 하이라이트.
- **FR-5 근거화 요구가 곧 기능이 된다**: 날조 방어(QT-1)가 사용자에겐 "검증 가능한 요약"으로 보임 → 페르소나가 가장 불신하는 "재현 안 되는 논문" 맥락에서 신뢰 확보.

## 8. 비용

- **건당 비용(대략)**: 논문 1편 — 요약 Sonnet(~13K입력+~1K출력) ≈ **$0.1~0.2** · 초록 번역 Haiku(~350입력+~500출력) ≈ **$0.003** · **전문 번역 Haiku**(전문 ~13K입력 + 한국어 ~18K출력; 한국어 ~1.3배) ≈ **$0.1**(보통)~**$0.3**(장문 ~40K). *(전문 번역 = 초록 번역의 ~30~100배지만 절대값은 요약과 비슷, 온디맨드+영구저장으로 bounded.)*
- **과금 단위 = "열어본 distinct 논문 × 1회"**: 온디맨드 + 영구저장 → 전체 코퍼스가 아니라 *실제 조회된* 논문만, 그것도 1회. 재조회·타 사용자는 공짜 → **요약 기본 Sonnet도 bounded.**
- **이중 호출 낭비 없음**: 요약은 처음부터 Sonnet, 번역은 Haiku → 단일 모델 1콜. 게이트 실패는 *같은 모델 1회 재시도*(승격 아님).
- **map-reduce 아웃라이어**: 초장문은 콜 수↑ → 입력 토큰 캡으로 상한.
- ⚠️ **예산 미계상 주의**: 요약 Sonnet은 인프라 비용표(`infrastructure-design.md` §6, Bedrock 임베딩 $5-10/월만 계상)에 **없는 신규 비용 라인**. NFR-C1 $1,600 상한엔 여유(현재 ~$370-400 사용)지만, **U7 도입 시 비용표에 별도 계상** 필요.

## 9. 용어집 & 개인화 (확정 설계)

### 9.0 핵심 분기선 — 공유 / 프리셋 / per-user

개인화를 비용 구조로 가르면 **네 종류** — 핵심은 "추가 생성이 드나":

| 구분 | 비용 | 예 | 결론 |
|---|---|---|---|
| **공유(객관)** | 1회 생성·영구저장·이후 공짜 | 기본 요약·번역·미번역 용어 | 무료 |
| **persona 생성 변형** | 논문당 최대 2벌(전문/입문) — bounded | 전문가용/입문자용 요약 | 무료(코어) |
| **뷰 프리셋** | **추가 생성 0** — 같은 출력의 표시 변형 | 3줄 핵심·관점별 강조 | 무료 |
| **진짜 per-user(자유 입력)** | 사용자마다 유일 → 사용자 수에 비례 | "내 연구노트와의 관련성" | 본 문서 밖 → #9 |

> **요지**: 본 문서의 개인화는 전부 **싸다** — persona는 2벌 생성, 나머지 프리셋은 뷰라 추가비용 0 → **등급 구분 없이 그냥 제공(확정)**. 비용이 사용자 수에 비례하는 *진짜* per-user(자유 입력 연구노트)만 #9로 빠졌다. (과금 모델은 본 설계 범위 밖.)

### 9.1 [채택] 용어 레벨 — 용어집(Glossary) [공유+개인]

> **도입**: 미번역 리스트(P1)·용어 선호 저장(P2) 포함 — 싸고 효과 큼.

학술 번역 최대 약점 = **용어 일관성**("attention"→"어텐션" vs "주의").

| 단계 | 내용 | 성격 |
|---|---|---|
| **P1 시드 도메인 용어집** | AI/ML 표준 용어맵(고정). **「미번역(keep-as-is)」 리스트 포함** — Transformer·BERT·LoRA·RAG 등 핵심 용어는 영어 그대로 | 공유·싸고 안전 |
| **P2 개인 오버라이드** (DeepL식) | 유저가 용어 수정 → 개인 용어집 저장 → 이후 강제. UI는 **"이 번역 선호 저장" 버튼** 한 개면 충분 | 개인 |
| **P3 커뮤니티 공유 용어집** | 집단 교정 공유 | ⚠️ UGC → 모더레이션·콘텐츠 삽입 방어(Part 2-B). **제외**(별개 기능, §12) |

- **미번역 리스트(P1)가 가성비 최고**: 모델명·약어를 영어로 고정하는 것만으로 번역 만족도 급상승, 비용 0.
- **피드백은 term 레벨로**: 전체 번역문을 자유 편집·few-shot 참고하는 방식은 일관성·비용이 불리. "잠재공간→latent space" 같은 **용어 매핑**으로 좁히면 robust(80/20).
- **저비용 교정**: 단순 명사 교체는 캐시된 기본 번역에 **결정적 후치환**으로 LLM 재호출 회피.
- **피드백 루프**: "용어 수정/선호 저장" → 개인 용어집 upsert → `glossaryVer++` → 해당 사용자 키 갱신.

### 9.2 개인화 — 스타일·관점 (확정 설계)

> 핵심 통찰 — **요약은 한 번만 풍부하게 생성하고, 대부분의 "프리셋"은 그 구조화 출력을 *다르게 보여주는 뷰*다 → 추가 LLM 0.**

**① 생성 변형** (내용이 실제로 달라짐 → 생성·캐시): = `persona` 파라미터(코어, stage 6). 논문당 최대 2벌.

| 수준 | 규칙 |
|---|---|
| **expert** | 전문용어 유지 · 약어 유지 · 수식 유지 |
| **beginner** | 전문용어·약어는 **영문 원어 그대로 표기**(한글 음차/치환 금지) + **첫 등장 시 한국어 괄호 풀이** · 약어 첫 등장 시 원문 전개 · 수식은 자연어 해설 추가 · **이후 등장은 원어(영문) 유지** (2026-06-19 구체화) |

**② 표시 프리셋** (같은 §3 구조화 출력의 *뷰* → 추가 LLM·캐시 0):
| 프리셋 | 보여주는 것 |
|---|---|
| **전체**(기본) | 전 필드 |
| **3줄 핵심** | `tldr`만 |
| **성능 관점** | `results` 강조(benchmark·dataset·metric) |
| **구현·수식 관점** | `method` 강조(architecture·hyperparam) |
| **리뷰어 관점** | `contributions`+`limitations` 강조 |

- 뷰는 §3 JSON에서 **클라이언트가 필드 선택 렌더** → 재생성·추가비용 없음, `styleVer`로 캐시 폭발 없음.
- 따라서 개인화 = **persona(2벌 생성) + 무료 뷰 프리셋**으로 **완결**. (자유 입력 "내 연구노트 관련성"만 진짜 per-user → #9.)

## 11. 저장 & 캐시 전략 — 영구 저장 + 핫캐시 2단

핵심: 키가 `(paperId, version, task, lang, scope, persona, glossaryVer, modelVer, promptVer)`로 **immutable** 이므로, 같은 키의 결과는 항상 동일. 따라서 **"7일 후 버리는 캐시"가 아니라 "영구 저장"이 옳다** — (논문 × 작업 × 수준) 평생 1회만 생성. **요약: persona∈{expert,beginner}·scope=full / 번역: persona=none·scope∈{abstract,full}** — 각 조합이 별개 객체(2026-06-19 개정).

| 계층 | 저장소 | 역할 | 만료 |
|---|---|---|---|
| **영구(진실원본)** | **S3** | (paper, model, prompt, …) 당 1회 생성 → 영구 보관. 전문 저장 패턴 재사용, 신규 인프라 0 | 없음(키 변경 = 신규 객체) |
| **핫캐시** | **ElastiCache Redis**(기존 자산) | 자주 보는 논문 빠른 응답 | 짧은 TTL(예: 24h) |

- **읽기(read-through)**: Redis → 미스 → S3 → 미스 → 생성 → S3+Redis write-through.
- **객체 경로 예**: `summaries/{paperId}/v{version}/{task}_{lang}_{scope}_{persona}_{modelVer}_{promptVer}.json`
- **무효화**: modelVer/promptVer/glossaryVer/version 변경 → **키(경로)가 바뀌어** 자동으로 새 객체 생성. 옛 객체는 미참조로 방치(또는 라이프사이클 정리). 수동 flush 불필요.
- **신선도**: 영구 저장이어도 stale 아님 — 모델/프롬프트 업그레이드 시 키가 바뀌어 자동 갱신.
- **스토리지 부담**: 요약 ~2KB × 30만 편 ≈ ~600MB → S3에선 무시 수준.

## 12. 설계 결정 (확정) · 외부 의존

**확정 (측정 불필요 — 바로 이렇게 짠다):**
- **모델**: 요약=Sonnet, 번역=Haiku. **고정.**
- **번역 범위**: 초록(`scope=abstract`) + **전문(`scope=full`)** 둘 다. 기본 abstract, 사용자가 전문 번역 별도 트리거(2026-06-19 개정).
- **용어집**: P1(시드+미번역 리스트)·P2(개인 선호 저장) **포함**. P3(커뮤니티 공유)는 **제외**(모더레이션 필요 = 별개 기능).
- **개인화**: **요약은 persona(전문/입문) 2벌** + 무료 뷰 프리셋(§9.2). **번역은 단일**(충실성 작업, persona 부적합 → 입문자 친화는 용어집). (2026-06-19 개정)
- **동기/비동기**: 기본 **스트리밍 동기**, 초장문(map-reduce)만 **비동기 잡**. 확정.
- **품질 게이트**: **앵커 실재성 + 스키마** 결정적 체크만. **LLM-judge 안 씀**(콜 2배).
- **재현성 신호**: 같은 요약 호출에서 **정규식 힌트 + LLM 추출** 병행. 확정.

**외부 의존 (내 설계 밖, 프로세스/타 기능):**
- **신규 유닛 등재**: 요약/번역을 요구사항(U7)에 정식 등재 — frozen 범위 밖이라 **승인 절차** 필요.
- **U6 grounding 재사용 검증**: `GroundingEnforcementHook`이 검색용으로 설계됨 → 요약 주장에 그대로 쓰이는지 U6 FD 확인 필요.
- **#9·#12 재사용**: 본 파이프라인 6~7단계(LLM+근거화)를 차별화/근거형성 에이전트 노드로 재사용 — 앞에 "유사논문 검색" 노드만 추가.

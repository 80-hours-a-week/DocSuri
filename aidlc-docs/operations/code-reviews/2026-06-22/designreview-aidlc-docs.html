HIGH U2 SearchOrchestrationService Calls CostGuardCircuitBreaker Synchronously on Hot Path (critique)
The design specifies that U2.SearchOrchestrationService calls U6.CostGuardCircuitBreaker.getBudgetState() synchronously on every search request (NFR-P1 P50<3s target). While getBudgetState() is described as returning a 'near-real-time' cached state, the synchronous dependency on a cross-unit service on the critical latency path creates a coupling risk. If the CostGuard service has any latency spikes, lock contention on budget state updates (recordSpend + evaluateCircuit racing with getBudgetState), or deployment issues, it directly degrades the P50<3s SLA for all search requests. The design does not specify a timeout for this call, a fallback behavior if getBudgetState() is unavailable, or what the circuit state is during U6 startup/restart.
HIGH GroundingEnforcementHook Applied Post-Handler Creates Unbounded Latency Tail on P50<3s Path (critique)
The design places GroundingEnforcementHook.enforce() in the response edge post-handler of U6.ApiGatewayMiddleware, applied synchronously after the full U2 pipeline completes. The enforce() method performs: (1) candidate response to retrieved record set mapping verification, (2) AI text source verification against retrieved records, and (3) violation detection. This full enforcement occurs on every search response within the P50<3s SLA. The design does not specify a timeout for enforce(), the computational complexity of the verification algorithm, or what happens if the retrieved record set is large. For degraded mode (LEXICAL_ONLY with potentially large BM25 result sets), this verification cost may be higher. The runEvalSet path reuses the same hook, meaning eval set execution characteristics inform the production path's expected latency.
HIGH SearchExecutedEvent at-least-once Delivery Without Idempotency Key Collision Specification (critique)
The design specifies that SearchHistoryService.recordSearch() achieves idempotency via dedupe_key = sha256(owner_id | executed_at.isoformat() | query). However, the SearchExecutedEvent is published by SearchOrchestrationService after a successful response using timestamp from the request context. Two scenarios create correctness risks: (1) If a search completes and publishes the event but the response is lost to the client (network fault), the client may retry the search, generating an identical event with the same timestamp+query, which would correctly deduplicate—but if the retry occurs within the same second and the system clock resolution means timestamps are equal, this is correct. However if the retry is treated as a new search by the system (new requestId), the timestamp may differ by milliseconds, producing a different dedupe_key and creating a duplicate history entry. (2) The timestamp used in the dedupe_key appears to be SearchExecutedEvent.timestamp (search execution time), but if the event backbone delivers the event after a delay, the consumer's 'now' time differs from the timestamp in the event. The design does not clarify whether 'timestamp' in the event is the time of search execution (producer-set) or delivery time.
HIGH U4 rerun Via SearchGatewayPort Has No Specification for Authentication Context Propagation (critique)
The design specifies that U4 SavedSearchService.rerun() and SearchHistoryService.rerun() call SearchGatewayPort.search(query, principal) which re-enters U6.ApiGatewayMiddleware → U2 pipeline. However, the rerun originates from within the API server process (not from a fresh HTTP request), meaning there is no incoming HTTP request with session cookie. The design is silent on: (1) how the principal/auth context is serialized into the re-entry call, (2) whether the gateway middleware's authn/authz enforcement (which normally reads from HTTP request headers/cookies) is bypassed, re-executed, or substituted by the injected principal, (3) whether rate limiting counts the rerun as a separate request for the same user (it should), and (4) how the requestId chain is preserved for observability. The StubSearchGateway implementation placeholder does not address these concerns, deferring them to 'U6/Infra' without design specification.

High Findings (4)

HIGH U2 SearchOrchestrationService Calls CostGuardCircuitBreaker Synchronously on Hot Path ▶

Location: component-dependency.md §2 U2 Discovery, services.md SearchOrchestrationService, services.md GatewayPipelineService, ports.md §3 CostGuardCircuitBreaker

Description: The design specifies that U2.SearchOrchestrationService calls U6.CostGuardCircuitBreaker.getBudgetState() synchronously on every search request (NFR-P1 P50<3s target). While getBudgetState() is described as returning a 'near-real-time' cached state, the synchronous dependency on a cross-unit service on the critical latency path creates a coupling risk. If the CostGuard service has any latency spikes, lock contention on budget state updates (recordSpend + evaluateCircuit racing with getBudgetState), or deployment issues, it directly degrades the P50<3s SLA for all search requests. The design does not specify a timeout for this call, a fallback behavior if getBudgetState() is unavailable, or what the circuit state is during U6 startup/restart.

Recommendation: Define an explicit timeout for getBudgetState() calls (e.g., <5ms). Specify a safe fallback when the call fails or times out (e.g., assume NORMAL mode to preserve availability). Consider whether CostGuard state should be pushed to U2 via a shared in-process cache refreshed by a background loop rather than pulled on each request, which would eliminate the synchronous cross-unit dependency on the hot path while maintaining near-real-time accuracy.

HIGH GroundingEnforcementHook Applied Post-Handler Creates Unbounded Latency Tail on P50<3s Path ▶

Location: services.md GatewayPipelineService, component-dependency.md §6 U6, component-methods.md U6 GroundingEnforcementHook, ports.md §2

Description: The design places GroundingEnforcementHook.enforce() in the response edge post-handler of U6.ApiGatewayMiddleware, applied synchronously after the full U2 pipeline completes. The enforce() method performs: (1) candidate response to retrieved record set mapping verification, (2) AI text source verification against retrieved records, and (3) violation detection. This full enforcement occurs on every search response within the P50<3s SLA. The design does not specify a timeout for enforce(), the computational complexity of the verification algorithm, or what happens if the retrieved record set is large. For degraded mode (LEXICAL_ONLY with potentially large BM25 result sets), this verification cost may be higher. The runEvalSet path reuses the same hook, meaning eval set execution characteristics inform the production path's expected latency.

Recommendation: Define an explicit timeout budget for enforce() within the overall P50<3s SLA (e.g., ≤200ms). Document the algorithmic complexity of anchor/source verification relative to result set size. Consider whether the full grounding check needs to complete synchronously before response delivery, or whether a fast structural check (schema completeness, non-empty anchors) can gate delivery while deeper verification runs asynchronously and triggers a retrospective incident if violations are found. Add a circuit breaker around the enforce() call itself so grounding failures degrade gracefully (abstain) rather than causing latency spikes.

HIGH SearchExecutedEvent at-least-once Delivery Without Idempotency Key Collision Specification ▶

Location: shared/events.md §2 SearchExecutedEvent, U4 business-logic-model.md §6, component-methods.md SearchHistoryService.recordSearch

Description: The design specifies that SearchHistoryService.recordSearch() achieves idempotency via dedupe_key = sha256(owner_id | executed_at.isoformat() | query). However, the SearchExecutedEvent is published by SearchOrchestrationService after a successful response using timestamp from the request context. Two scenarios create correctness risks: (1) If a search completes and publishes the event but the response is lost to the client (network fault), the client may retry the search, generating an identical event with the same timestamp+query, which would correctly deduplicate—but if the retry occurs within the same second and the system clock resolution means timestamps are equal, this is correct. However if the retry is treated as a new search by the system (new requestId), the timestamp may differ by milliseconds, producing a different dedupe_key and creating a duplicate history entry. (2) The timestamp used in the dedupe_key appears to be SearchExecutedEvent.timestamp (search execution time), but if the event backbone delivers the event after a delay, the consumer's 'now' time differs from the timestamp in the event. The design does not clarify whether 'timestamp' in the event is the time of search execution (producer-set) or delivery time.

Recommendation: Clarify that SearchExecutedEvent.timestamp is producer-set (search execution time, not delivery time) and that the dedupe_key must use this producer-set timestamp. Add a unique search requestId to the SearchExecutedEvent schema to serve as the primary idempotency key, reducing reliance on the timestamp+query composite. Document the clock resolution assumption (e.g., timestamps are millisecond precision) and whether two searches of the same query by the same user within that resolution window should be treated as one or two history entries.

HIGH U4 rerun Via SearchGatewayPort Has No Specification for Authentication Context Propagation ▶

Location: U4 business-logic-model.md §5, business-rules.md BR-L9, component-dependency.md §4, services.md SavedSearchService and SearchHistoryService

Description: The design specifies that U4 SavedSearchService.rerun() and SearchHistoryService.rerun() call SearchGatewayPort.search(query, principal) which re-enters U6.ApiGatewayMiddleware → U2 pipeline. However, the rerun originates from within the API server process (not from a fresh HTTP request), meaning there is no incoming HTTP request with session cookie. The design is silent on: (1) how the principal/auth context is serialized into the re-entry call, (2) whether the gateway middleware's authn/authz enforcement (which normally reads from HTTP request headers/cookies) is bypassed, re-executed, or substituted by the injected principal, (3) whether rate limiting counts the rerun as a separate request for the same user (it should), and (4) how the requestId chain is preserved for observability. The StubSearchGateway implementation placeholder does not address these concerns, deferring them to 'U6/Infra' without design specification.

Recommendation: Define the SearchGatewayPort interface contract to explicitly include: authenticated principal passing (not just the query), a mechanism for the gateway to validate the injected principal without re-doing session cookie lookup, rate limit counting for rerun requests, and requestId propagation. Consider whether the gateway re-entry should be implemented as an internal function call with principal injection (bypassing HTTP transport entirely) rather than a literal HTTP re-request, which would eliminate the auth context serialization problem while still applying all middleware business logic.

Medium Findings (5)

MEDIUM Single Event Bus with No Dead Letter Queue Specification for SearchExecutedEvent ▶

Location: shared/events.md §2, U4 business-logic-model.md §6, services.md SearchHistoryService

Description: The design specifies DLQ for U1 ingestion failures (IngestFailureHandler.sendToDLQ) and for AI incident events, but does not specify DLQ handling for SearchExecutedEvent consumption failures in U4.SearchHistoryService. The at-least-once guarantee means if the event bus delivers the event but U4 fails to process it (e.g., database unavailable, dedupe key computation error), the event may be retried indefinitely or dropped depending on the event bus configuration—neither of which is documented. Given that search history is a user-facing feature (FR-10, US-L3), silent history loss or indefinite retry storms should be addressed.

Recommendation: Define a DLQ for SearchExecutedEvent consumption failures with a maximum retry count and poison message handling. Document the acceptable history loss tolerance (e.g., history may be missing for events that exhaust retries) and surface this as a known limitation in the operational runbook. Since history is non-critical compared to search functionality, a simple drop-after-N-retries with telemetry emission is likely acceptable and should be explicitly stated.

Description: The U8 CitationSnapshot entity has a 'createdAt' field described as 'TTL 판단 기준' (TTL determination basis), and BR-CG9 (Manual Refresh) states that manual refresh attempts provider re-query even before TTL expiry. MEDIUM VectorSpec as Shared Contract Has No Change Management Process ▶

Location: shared/ports.md §1, U1 domain-entities.md §8 VectorSpec, U2 domain-entities.md §9, U2 business-rules.md §6, unit-of-work.md shared/ vector-spec

Description: The design establishes VectorSpec (dimensions, modelRef, distanceMetric) as a critical immutable shared contract between U1 (writer) and U2 (reader), with the statement 'VectorSpec 변경=전체 재임베딩(단방향)' (VectorSpec change = full re-embedding, one-way). The design notes that NFR Requirements PIN the values. However, there is no documented process for what happens if the embedding model must be changed (e.g., model deprecation, quality improvement): the full re-embedding procedure, the dual-read period during transition, how U2 queries are served during re-indexing, or how the specVersion mismatch is detected and handled at runtime. The design mentions 'specVersion 일치' as a requirement but not how specVersion mismatch is detected or handled.

Define specVersion validation at U2 startup: if the VectorSpec in the index does not match the configured spec, U2 should refuse to serve queries rather than return silently wrong results. Add a runtime specVersion check in HybridRetriever.

Alternative Approaches

Alternative 1: Current Approach — Modular Monolith API + Separate Workers with Shared Ports for Cross-Cutting Concerns ▶

The current design is a DQ1 modular monolith (U2/U3/U4/U7/U8 as in-process modules under a single API deployment) with a separate U1 ingestion worker and U6 ops worker. Cross-cutting concerns (grounding, cost, observability) are owned by U6 and exposed via shared/ports interfaces using dependency inversion, preventing compile-time circular dependencies. The GroundingEnforcementHook fires synchronously in the U6 gateway post-handler on every search response, and CostGuardCircuitBreaker.getBudgetState() is called synchronously on every search request. Search history writes are decoupled via a SearchExecutedEvent on the event backbone. The design is architecturally ambitious, with careful attention to single-authority ownership rules (grounding, authorization, cost) and deep security layering.

What Changes: This is the baseline. No changes.

Implementation Complexity: High — Eight units with complex inter-unit contracts, dual sync/async communication patterns, shared port interfaces requiring careful DI wiring, and multiple unresolved specification gaps (REDACTED tokens, missing getFullText API, unspecified rerun auth propagation) that will surface as integration-time surprises.

Advantages:

  • Strong single-authority ownership rules (U3 for authz, U6 for grounding/cost) prevent double-enforcement and divergence across units
  • Dependency inversion via shared/ports prevents U2↔U6 compile-time circular dependency while still allowing runtime injection
  • Modular monolith simplifies deployment (single API process) while maintaining logical separation, appropriate for a demo-scale system
  • Event-driven history write (SearchExecutedEvent) correctly removes history recording from the P50<3s critical path
  • Deep security layering (gateway pre/post chain, owner-scoped repositories as data backstop, U3 AuthorizationGuard as single decision point) provides defense in depth

Disadvantages:

  • GroundingEnforcementHook.enforce() runs synchronously post-handler on every search response with no specified timeout or complexity bound, creating an unbounded latency tail risk on the P50<3s SLA path
  • getBudgetState() called synchronously on every search with no fallback if CostGuard is unavailable during startup/restart/deployment, tightly coupling search availability to cost guard availability
  • U4 rerun via SearchGatewayPort has no specification for how auth context (principal, session cookie) is propagated from an in-process call back through the gateway middleware chain that normally reads HTTP headers
  • Multiple REDACTED_SECRET tokens in the design documents make key components (U4 sub-repositories, U6 ObservabilityService orchestration, U2 HybridRetriever, U1 IngestionResilienceService) unverifiable during review
  • U7 frontend depends on a getFullText backend API that has no corresponding backend FD, controller, or service specification, leaving US-S3 (Source View) unimplementable at integration time
  • VectorSpec change management process is entirely unspecified despite the design acknowledging that any change requires full corpus re-embedding
Alternative 2: Async-First Search Pipeline with Pre-Computed Budget Cache and Offline Grounding Validation ▶

This alternative addresses the two highest-severity latency risks in the current design by restructuring the synchronous hot path. Rather than calling getBudgetState() and running GroundingEnforcementHook.enforce() synchronously on every search request within the P50<3s window, the budget state is pushed as a server-sent update to a local in-memory cache in the search module (refreshed every 5 seconds via a background loop), and grounding enforcement is restructured as a post-response async validation that can block result display on the frontend rather than blocking the server-side response. The search response returns immediately with a 'grounding-pending' token, and the frontend polls or receives a push update once grounding completes. This decouples the two most latency-sensitive cross-cutting concerns from the synchronous request-response path while preserving the single-authority ownership rules.

What Changes: CostGuardCircuitBreaker gains a push-subscription interface alongside getBudgetState(); U6 publishes BudgetState updates to an in-process subscriber in U2's SearchOrchestrationService, which caches the last known state with a staleness TTL (e.g., 10s). If cache is stale/unavailable, U2 defaults to NORMAL mode (fail-open for cost, not fail-closed — acceptable tradeoff for search availability). GroundingEnforcementHook.enforce() is extracted from the synchronous post-handler into an async validation step: the gateway returns SearchResultPageDTO immediately with a groundingStatus:'pending' field, and a separate GroundingValidationWorker processes the result and publishes a GroundingCompletedEvent. U5 ApiClient either polls GET /api/search/{requestId}/grounding or receives a WebSocket/SSE push. The grounding result then replaces the pending state client-side. Grounding violations still result in AbstainDTO — the frontend transitions from 'pending' cards to the abstain state. U6 GroundingEnforcementHook remains the single authority; only the invocation site moves from sync post-handler to async worker.

Implementation Complexity: High — Requires new async validation worker, new client-polling or push infrastructure, new 'pending' frontend state handling, in-process subscription mechanism for BudgetState, and careful specification of what 'grounding-pending' means for FR-5 compliance — substantially more moving parts than the current synchronous approach.

Advantages:

  • Eliminates unbounded enforce() latency from the P50<3s critical path — search responses return as soon as U2 pipeline completes, with grounding applied asynchronously
  • BudgetState local cache eliminates the synchronous cross-unit dependency on CostGuard availability; search continues with last-known state during CostGuard restarts/deployments
  • Fail-open for budget state (default NORMAL when cache stale) is a defensible tradeoff for a demo system — worst case is one overspend interval, not search unavailability
  • Grounding single-authority rule (U6 owns enforce()) is fully preserved — only the invocation pattern changes from sync to async

Disadvantages:

  • Async grounding creates a 'pending cards' UX state that does not exist in the current design — U5 must handle a new intermediate state (grounding-pending → grounded or abstained), adding frontend complexity
  • Polling or SSE/WebSocket infrastructure for grounding completion adds new protocol complexity to U5 ApiClient and the backend gateway, which is significant overhead for a demo system
  • The window between returning pending cards and completing grounding validation means users may briefly see card titles/snippets before an abstain decision, which could be considered a grounding violation depending on interpretation of FR-5
  • Budget state fail-open (defaulting to NORMAL when cache is stale) contradicts the design's general fail-closed philosophy and could allow LLM cost overruns during CostGuard outages
Alternative 3: Simplified Gateway with Bounded Enforce() Contract and Explicit Fallback Specifications ▶

Rather than restructuring the architecture, this alternative directly addresses the critical specification gaps in the current design while preserving the synchronous architecture. The core insight is that most of the high-severity constraints (unbounded enforce() latency, missing getBudgetState() fallback, unspecified rerun auth propagation, missing getFullText API) are specification deficiencies rather than architectural flaws. This alternative adds explicit bounded contracts to the existing components: enforce() gets a hard timeout with a pass-through fallback (fail-open for grounding on timeout, but with an incident signal), getBudgetState() gets a local in-memory default (NORMAL) returned if the call exceeds a latency budget, the SearchGatewayPort.search() rerun path gets an explicit InternalSearchRequest wrapper that carries an already-authenticated principal and bypasses cookie-based authn, and the getFullText API gap is closed by adding a PaperFullTextController to U7 backed by the existing FullTextSourceAdapter. The architecture remains unchanged; only the contracts become complete.

What Changes: GroundingEnforcementHook.enforce() contract adds: enforceTimeout (e.g., 200ms), onTimeout behavior (pass=true with GroundingDecision{verdict:pass, violations:[], timedOut:true}), and a HallucinationDetector signal on timeout for observability. CostGuardCircuitBreaker.getBudgetState() contract adds: callTimeout (e.g., 50ms), fallbackState (BudgetState{tier:NORMAL, degradeMode:NORMAL, circuitState:CLOSED}), and a CostExplosionDetector signal if fallback is used. SearchGatewayPort gains an InternalSearchRequest type wrapping (query, authenticatedPrincipal, parentRequestId) that bypasses HTTP cookie authn in the gateway middleware — U6.AuthnAuthzGuard checks for pre-authenticated principal in the request context before attempting session verification. U7 gains PaperFullTextController (GET /api/papers/{paperId}/fulltext), PaperFullTextService (reads StoredFullText from ObjectStorage via FullTextSourceAdapter with OA license check), and corresponding response DTOs. U4 SearchHistoryService.recordSearch() gains a DLQ-equivalent: failed events after N retries are written to a HistoryDeadLetterStore for operator inspection rather than silently dropped. U3 gains EmailVerificationToken as a first-class entity in domain-entities.md with cleanup via a scheduled RefreshScheduler-style trigger.

Implementation Complexity: Medium — The changes are additive specifications to existing components rather than architectural restructuring. The most complex change (InternalSearchRequest for rerun auth) follows existing patterns already established in the gateway middleware. Updating frozen port signatures requires coordination but not redesign.

Advantages:

  • Preserves the current synchronous architecture with its well-understood latency characteristics and deployment simplicity — no new async patterns, polling infrastructure, or frontend state machines
  • Bounded enforce() timeout with pass-through fallback ensures P50<3s SLA is protected even if grounding validation is slow, while the timedOut signal preserves observability and incident detection
  • Explicit InternalSearchRequest for rerun auth propagation closes a concrete integration gap (constraint ec42c41a) with minimal design change — the gateway middleware already has a principal injection path
  • Adding PaperFullTextController to U7 closes the getFullText API gap (constraint ed27c8ab) with a self-contained change that follows the existing FullTextSourceAdapter pattern in U7
  • Fails-open rather than fails-closed for cost and grounding on timeout — appropriate tradeoff for a demo system where search availability is more valuable than perfect enforcement

Disadvantages:

  • Fail-open for grounding on enforce() timeout means some potentially ungrounded results may surface to users during high-load or slow verification scenarios, which may conflict with strict FR-5 interpretation
  • Adding explicit timeout and fallback contracts to existing components requires updating shared/ports.md frozen signatures (enforce() and getBudgetState() are marked FROZEN), which has wide coordination impact across U2, U6, and their FD documents
  • The REDACTED_SECRET tokens in the design documents cannot be resolved by this alternative — the architectural assessment remains incomplete until those identifiers are restored, and this alternative assumes they represent legitimate component names
Alternative 4: Event-Sourced Search History with Materialized View and Explicit VectorSpec Migration Protocol ▶

This alternative addresses two medium-severity constraints (SearchExecutedEvent idempotency ambiguity and VectorSpec change management) by applying two targeted architectural patterns. For search history, it replaces the current sha256-based dedupe key with an event-sourced approach: SearchExecutedEvent carries a client-generated idempotency key (UUID generated by U2 at search execution time, embedded in the event), making the dedupe logic explicit and eliminating the timestamp-resolution ambiguity identified in constraint aa34e9fe. For VectorSpec, it introduces a versioned dual-index migration protocol with a specVersion field on every IndexRecord, allowing U1 to write new embeddings alongside old ones during a migration window while U2 continues to serve from the old index, with a cutover flag controlled by U6 that switches U2's read target atomically once re-indexing is complete. These two changes are independent and can be adopted separately, but together they close two significant correctness risks in the current design.

What Changes: SearchExecutedEvent gains an idempotencyKey field (UUID v4, generated by SearchOrchestrationService.publishSearchExecuted() at event creation time, included in the event payload alongside userId/query/timestamp/resultCount). U4.SearchHistoryService.recordSearch() replaces dedupe_key=sha256(owner_id|executed_at_iso|query) with dedupe_key=event.idempotencyKey — simple, unambiguous, and immune to timestamp resolution issues. The events.md SearchExecutedEvent FROZEN schema is updated to add idempotencyKey as a required field (this requires a coordinated schema change per the evolution policy). For VectorSpec migration: IndexRecord gains a specVersion field (integer, default 1). VectorIndexWriter.upsert() writes to a versioned index partition (e.g., index_v1, index_v2 during migration). U6 gains a VectorSpecMigrationController (internal ops endpoint) that controls a migration flag: STABLE (U2 reads from current specVersion index), MIGRATING (U1 writes to both old and new index, U2 reads from old), CUTOVER (U2 switches to new index atomically). HybridRetriever.retrieve() reads the specVersion from U6's migration flag via a new VectorIndexVersionPort. RefreshScheduler.triggerFullRebuild() gains a targetSpecVersion parameter to initiate versioned re-indexing. The VectorSpec entity in shared/vector-spec gains a specVersion integer and migration status.

Implementation Complexity: Medium — The idempotency key change is low complexity (add one field, update one consumer). The VectorSpec migration protocol is medium complexity — it requires versioned index partitions, a migration state machine in U6, and updates to HybridRetriever — but follows established patterns and has clear bounded scope.

Advantages:

  • Explicit idempotency key on SearchExecutedEvent (UUID generated at publish time) eliminates the timestamp-resolution ambiguity in the current dedupe logic — two retried searches by the same user at the same second with the same query correctly produce only one history entry if they share an idempotencyKey (same request), or two entries if they are genuinely separate searches
  • Versioned dual-index migration protocol provides a concrete, safe path for VectorSpec changes (model deprecation, quality improvement) without requiring a search outage — U2 continues serving from the old index while U1 rebuilds the new one
  • The VectorSpecMigrationController gives operators explicit control over the cutover point, enabling validation of the new index quality before committing to the switch
  • Both changes are targeted and follow existing design patterns (event backbone, shared capabilities, ops dashboard) without introducing new architectural layers

Disadvantages:

  • Updating the FROZEN SearchExecutedEvent schema to add idempotencyKey requires a coordinated schema change across U2 (producer) and U4 (consumer) with a migration window for existing events in-flight — the FROZEN status means this has explicitly wide impact
  • The dual-index write during migration doubles U1's write load and storage cost for the migration duration — for a corpus of hundreds of thousands of documents this could be significant in both time and cost
  • VectorSpecMigrationController introduces a new operational concern (someone must monitor migration progress, validate index quality, and trigger cutover) that requires runbook documentation and operator training not currently scoped in the design

Recommendation

Alternative 3 (Simplified Gateway with Bounded Enforce() Contract and Explicit Fallback Specifications) is the most pragmatic choice for this project. The current design (Alternative 1) has genuine architectural merit — the single-authority ownership rules, dependency inversion via shared/ports, and modular monolith structure are all well-suited to a demo-scale AI/ML discovery system. However, the two high-severity latency risks (unbounded enforce() on the hot path, synchronous getBudgetState() with no fallback) and the concrete integration gap (missing getFullText API for US-S3) are specification deficiencies that will cause real failures at integration time rather than architectural flaws requiring restructuring. Alternative 3 closes these gaps with the minimum viable changes: bounded timeouts with explicit fallback contracts on enforce() and getBudgetState(), an InternalSearchRequest wrapper for rerun auth propagation, PaperFullTextController to close the US-S3 gap, and a DLQ path for SearchHistoryService. Alternative 2 (Async-First Pipeline) correctly identifies the latency risks but over-engineers the solution for a demo system — the 'grounding-pending' frontend state and SSE/polling infrastructure add substantial complexity for a correctness gain that can be achieved more simply with bounded timeouts. Alternative 4's targeted fixes (idempotency key on SearchExecutedEvent, VectorSpec migration protocol) are valuable and should be adopted alongside Alternative 3 rather than instead of it, particularly the idempotency key change which closes a genuine correctness risk with minimal coordination cost. The REDACTED_SECRET token issue identified in constraint 407c977e cannot be resolved by any architectural alternative — this is a documentation integrity problem that must be resolved by restoring the actual component names before any implementation begins, as the obscured identifiers (U4 sub-repositories, U2 HybridRetriever, U6 ObservabilityService orchestration) are referenced in critical data flow and traceability paths.

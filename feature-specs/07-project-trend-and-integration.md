# 07. 프로젝트 트렌드 & 통합 분석 (Project Trend & Integration Analysis)

> 사용자가 자기 프로젝트(코드 + 설계 문서 + README)를 시스템에 등록 → 시스템이 (a) 해당 프로젝트가 분야 트렌드를 따라가고 있는지, (b) 최신 논문의 어떤 기법을 통합할 수 있는지, (c) 어떤 수정점이 권장되는지 분석.
> **원본 #6 후반 + #7 병합.**

---

## 1. 핵심 요소

- **프로젝트 인입**: Git URL / 로컬 디렉터리 / README + 설계문서 텍스트. 코드 파일은 함수·모듈 단위 임베딩, 문서는 청크 임베딩.
- **프로젝트 프로파일**: (a) 사용 기법 추출 (코드 + README), (b) 도메인 분류, (c) 핵심 의도 문장 추출.
- **트렌드 비교 corpus**: 같은 분야 최근 1-3년 논문 N편.
- **분석 축**:
  - **트렌드 추종도**: 프로젝트가 사용하는 기법의 시기·인용 분포 vs 분야 전체 분포.
  - **통합 후보**: 프로젝트의 핵심 모듈에 대체/보완 가능한 최신 기법.
  - **권장 수정점**: 명백한 누락(예: 평가에서 표준 metric 미사용), 더 나은 baseline.
  - **방어 가능성**: 프로젝트의 차별점이 최신 트렌드 대비 여전히 유효한지.
- **anchor 양방향**: 권장사항은 (a) 프로젝트 코드/문서의 line/section, (b) 근거 논문의 anchor 둘 다 첨부.

---

## 2. 주요 문제

- **코드 임베딩 품질**: 텍스트 임베딩은 코드의 시맨틱을 부분적으로만 잡음. AST + 임베딩 하이브리드 필요.
- **프로젝트 의도 추출**: README/문서가 빈약하거나 outdated인 경우 시스템이 잘못된 의도로 분석.
- **트렌드 정의**: "트렌드"가 인용수인지 publication count인지 GitHub stars인지 — 측정 정의가 결과를 크게 흔듦.
- **권장의 "구체성"**: "transformer로 바꿔보세요" 같은 추상 제안은 무가치. 구체 파일·함수·라이브러리까지 짚어야 가치 있음.
- **프라이버시·IP**: 사용자 비공개 프로젝트가 시스템에 등록되면 IP 보호 필수.
- **시간적 정합성**: 프로젝트 코드와 비교 corpus의 시기 정렬 (오래된 프로젝트를 최신 SOTA로 비판하면 부당).

## 3. 파이프라인 설계 & 기술 스택

```
프로젝트 등록 (Git URL / files / docs)
    │
    ▼
┌─────────────────────────┐
│ Project Ingestor        │
│ - 코드: AST + 임베딩    │
│ - 문서: 청크 임베딩      │
│ - 메타: README 파싱      │
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Project Profiler (LLM)  │
│ - 도메인 분류            │
│ - 사용 기법 리스트       │
│ - 핵심 의도 진술         │
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Trend Corpus Selector   │  도메인에 맞는 최근 N편 검색
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Trend Analyzer          │
│ - 기법 분포 비교         │
│ - 메트릭 표준화 확인     │
│ - SOTA baseline 확인     │
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Integration Finder      │
│ - 프로젝트 모듈 × 후보   │
│ - 적합도 / 난이도 점수   │
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Recommendation Composer │  파일·줄·라이브러리·논문 anchor
│  (Claude Opus)          │
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Defensibility Checker   │  차별점이 여전히 유효한가
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Verifier                │  권장 ↔ 근거 논문 evidence
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Report UI               │  코드 diff 후보 + 논문 anchor
└─────────────────────────┘
```

### 기술 스택

| 레이어 | 선택 | 비고 |
|---|---|---|
| 코드 임베딩 | `voyage-code-3` 또는 CodeT5+ | 코드 전용 임베딩 |
| AST 파싱 | tree-sitter | 다언어 지원 |
| 코드 검색 | Qdrant 별도 collection | 논문과 분리 |
| Profiler | Claude Sonnet | README + 코드 요약 |
| Trend Analyzer | python + 통계 | 분포 비교 |
| Integration Finder | Claude Opus | 추론 깊이 |
| Defensibility | Claude Sonnet | 차별점 평가 |
| 시각화 | Mermaid (시계열 트렌드) | |

## 4. 차별화 포인트

- **자기 프로젝트 인지형 추천**: 일반 "최신 논문 추천"은 사용자 컨텍스트가 없다. 본 기능은 *사용자 프로젝트 코드/문서를 컨텍스트로* 사용해 "당신의 `model.py:120`을 X 논문의 §3 방법으로 바꿀 수 있다"처럼 짚어줌.
- **방어 가능성 평가**: 다른 도구가 안 하는 축. 사용자가 *왜 자기 프로젝트가 여전히 가치 있는지*를 확인. 대학원생 학위 방어 준비에도 활용.
- **anchor 양방향 (코드 + 논문)**: 권장사항이 코드 위치 + 논문 위치를 모두 가리키는 시스템은 드묾.
- **이전 분석과의 diff**: 같은 프로젝트를 3개월마다 분석 시, 트렌드 변화에 따른 권장 변화를 보여줌.

## 5. 위험 요소

- **IP 누출**: 비공개 프로젝트 등록 시 LLM 호출에 코드가 포함됨 → API provider의 데이터 정책 의존. Anthropic의 Zero Data Retention 옵션 + 사용자 동의 필수.
- **잘못된 권장의 비용**: "이 baseline으로 바꾸세요"가 틀리면 사용자가 무의미한 작업에 시간 투입. 모든 권장에 `effort 추정 + 위험도` 명시.
- **트렌드 추종 강요의 위험**: 모든 프로젝트가 SOTA를 따라야 하는 건 아님. 특히 응용 프로젝트는 안정성·검증된 기법이 정답. 사용자 의도(연구 vs 응용) 따라 권장 톤 변화 필요.
- **코드 임베딩 도메인 한계**: 도메인 특화 코드(예: 양자 회로, 통계학)에서 임베딩 품질 저하.
- **시기 정합성 누락**: 2020년 프로젝트를 2026 SOTA로 비판하면 무의미. 프로젝트 timestamp 확인 + 적정 시기 corpus 사용.
- **defensibility 평가의 LLM 한계**: 진짜 차별점은 종종 *암묵적 도메인 지식*. LLM이 "당신 프로젝트는 평범하다"고 잘못 판정.

## 6. 예상 비용

### 단위 비용 (프로젝트 1회 분석, 코드 ~50 파일 + 문서 ~30 페이지)

| 단계 | 비용 |
|---|---|
| Project 인입 (코드 임베딩) | ~$0.5 |
| Trend corpus 수집 (200편) | ~$1 |
| Profiler (Sonnet) | ~$0.3 |
| Trend Analyzer (룰+통계) | ~$0.1 |
| Integration Finder (Opus) | ~$2 |
| Recommendation Composer (Opus) | ~$2 |
| Defensibility | ~$0.5 |
| Verifier | ~$0.5 |
| **합계/분석** | **~$7** |

### MVP (월 200 분석)

- $7 × 200 = **$1,400/월**
- 같은 프로젝트 재분석 시 cache 적용 (~50% 절감) → 실제 **~$1,000/월**

### 스케일업 (월 5,000 분석)

- $7 × 5,000 = **$35,000/월**

> 단일 분석 비용은 06과 비슷하지만 *재분석 빈도*가 높음 (3개월마다 같은 프로젝트). incremental 분석 설계가 ROI 핵심.

# 04. 자동 모니터링 & 알림 (Auto Monitoring & Notification)

> 사용자가 등록한 관심 주제/쿼리를 스케줄러가 주기적으로 실행 → 신규 논문·인용 급증·새 SOTA·새 벤치마크 등 *"의미 있는 변화"* 만 필터링 → 사용자에게 알림.
> **원본 #4 (알림 서비스) + #9 (자동 검색 예약) 병합.**

---

## 1. 핵심 요소

- **쿼리 구독 등록 UI**: 사용자가 (a) 키워드/주제, (b) 주기(daily/weekly/monthly), (c) 알림 채널(email/web push/Slack/Discord), (d) 알림 수준(전체 신규 / 중요 변화만)을 등록.
- **스케줄러**: cron-like 작업으로 등록된 쿼리를 주기 실행. 사용자별로 격리.
- **상태 저장**: 각 구독의 "마지막 알림 시점의 상태 스냅샷"을 저장 (논문 ID 집합 + 인용수 + 벤치마크 결과).
- **변화 분류기**:
  - `new_paper`: 단순 신규 논문 (검색 결과에 처음 등장).
  - `citation_spike`: 기존 논문의 인용수가 지난 주기 대비 N% 또는 절대치 K 이상 증가.
  - `new_sota`: 특정 벤치마크의 SOTA를 갱신한 논문 (Papers with Code 데이터 연계).
  - `new_benchmark`: 새 벤치마크/데이터셋 발표 (키워드 기반 탐지).
- **요약형 알림**: 변화 항목을 한 줄 요약 + anchor + "왜 중요한지" LLM 설명 첨부.
- **알림 피로 제어**: 동일 사용자에 한 주기 최대 N건 (사용자 설정), 중요도 우선 정렬.

---

## 2. 주요 문제

- **"중요한 변화"의 정의**: SOTA·인용 급증의 임계값을 어떻게 정할지. 분야별로 베이스라인이 다름 (CV는 인용 폭증, niche 의학은 절대 인용수가 적음).
- **노이즈 비율**: arXiv는 하루 1000+ 신규. 키워드 매칭만으로는 70%가 무관 논문일 수 있음 → 임베딩 유사도 필터 추가 필요.
- **인용수 데이터 신선도**: Semantic Scholar의 citation count는 주 단위 업데이트. "급증"을 일 단위로 탐지하기 어려움.
- **벤치마크 데이터 가용성**: Papers with Code 외에는 벤치마크 SOTA를 구조화로 제공하는 곳이 거의 없음. NLP 외 분야는 자체 추출 필요.
- **사용자 컨텍스트 진화**: 사용자 관심이 시간에 따라 바뀌는데 등록 쿼리는 고정 → 자동 재학습 필요.
- **스케줄 충돌 & 부하**: 1만 명이 매일 09:00 알림이면 09:00 직전에 1만 쿼리 동시 발사 → rate limit 폭발.

## 3. 파이프라인 설계 & 기술 스택

```
사용자 구독 등록
    │
    ▼
┌─────────────────────────┐
│ Subscription Registry   │  Postgres: user_id, query, schedule, channel
└─────────┬───────────────┘
          │
          ▼
┌─────────────────────────┐
│ Scheduler (cron)        │  APScheduler / Celery beat / Temporal
└─────────┬───────────────┘
          │ (jittered to spread load)
          ▼
┌─────────────────────────┐
│ Query Executor          │  → 01. 검색 파이프라인 재사용
└─────────┬───────────────┘
          │ candidate list
          ▼
┌─────────────────────────┐
│ State Differ            │  vs last_snapshot
│ - new_paper             │
│ - citation_spike        │
│ - sota_update           │
│ - new_benchmark         │
└─────────┬───────────────┘
          │ change events
          ▼
┌─────────────────────────┐
│ Importance Classifier   │  LLM: 사용자 컨텍스트 + 변화 → 중요도 점수
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Digest Composer         │  N건 cap + 우선순위 정렬 + 한 줄 설명
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Notification Dispatcher │
│ - Email (SES/Postmark)  │
│ - Web push (FCM)        │
│ - Slack (incoming wh)   │
│ - Discord (webhook)     │
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ State Updater           │  새 snapshot 저장
└─────────────────────────┘
```

### 기술 스택

| 레이어 | 선택 | 비고 |
|---|---|---|
| 스케줄러 | Celery Beat (간단) / Temporal (복잡 워크플로/재시도) | Temporal은 retry·timeout 강력 |
| 상태 저장 | Postgres (구독·스냅샷) + Redis (실행 락) | 동일 잡 중복 실행 방지 |
| Citation 데이터 | Semantic Scholar API (`citationCount`) | 주 단위 신선도 한계 |
| SOTA 탐지 | Papers with Code API (NLP/CV 주력) + LLM 추출 (이외 분야) | |
| Importance 분류 | Claude Haiku | 변화 이벤트당 한 번 호출 |
| 알림 발송 | Amazon SES (email) + FCM (push) + Slack/Discord webhook | |
| Jitter 분산 | 사용자별 09:00 ± random(0, 30분) | rate limit 보호 |

## 4. 차별화 포인트

- **"중요한" 변화 분류**: 대다수 학술 알림(Google Scholar Alerts 등)은 신규 매칭만 보낸다. 본 시스템은 인용 급증·SOTA·새 벤치마크를 별도 카테고리로 분류 → 신호 대 잡음비 개선.
- **사용자 컨텍스트 인지형 importance**: LLM이 사용자의 등록 쿼리·구독 이력을 보고 "이 사용자에게 왜 중요한지" 설명 첨부.
- **다채널·다주기**: 일·주·월간 다른 주기 다른 채널 혼용 가능 ("긴급은 Slack, 주간 요약은 email").
- **알림 피로 자동 제어**: 사용자 클릭률을 학습해 자주 무시되는 카테고리는 다음 주기에 down-rank.

---

## 5. 위험 요소

- **알림 피로**: 매일 5건씩 받으면 30일 후 사용자 90%가 알림 해제. 첫 주에 "너무 많지 않은가" 자동 점검 + 사용자 빈도 조정 권장 메시지.
- **False positive SOTA**: LLM이 "SOTA 갱신"을 잘못 분류해 보내면 신뢰 무너짐. SOTA는 PwC 같은 검증 가능한 출처만 사용.
- **arXiv API 정책**: 대용량 자동 호출은 arXiv가 명시적으로 권장하지 않음 (bulk OAI-PMH 별도 인터페이스 사용 권장).
- **스케줄러 SPOF**: Celery beat 단일 인스턴스 다운 시 모든 사용자 알림 정지. Sentinel + healthcheck 필수.
- **이메일 reputation**: SES 신규 계정은 1일 발송 한도 200. 점진적 워밍업 필요. 워밍업 중에는 web push 우선.
- **GDPR/개인정보**: 사용자 구독 패턴은 연구 관심사 = 민감정보 가능. 익명화·삭제 권리 보장.

## 6. 예상 비용

### 단위 비용 (구독 1건 1회 실행)

| 단계 | 비용 |
|---|---|
| 검색 API | 무료 (학술 free tier) |
| 임베딩 reranker | ~$0.002 |
| State diff | DB I/O ~negligible |
| Importance 분류 (Haiku) | ~$0.001/이벤트 |
| 알림 발송 | email $0.0001, push $0 |
| **합계/구독/실행** | **~$0.005** |

### MVP (사용자 1,000명 × 평균 3구독 × 일 1회)

- 일 9,000 실행 × 30일 × $0.005 = **$1,350/월**
- 이메일 발송 SES: ~$30/월
- 인프라(스케줄러+DB+Redis): ~$80/월
- **총 ~$1,500/월**

### 스케일업 (사용자 10만 명 × 평균 5구독 × 일 1회)

- 일 50만 실행 × 30일 × $0.005 = **$75,000/월**
- 이메일 ~$1,500/월
- 인프라 ~$2,000/월
- **총 ~$78,500/월**

> 핵심 변동 요인: **사용자당 평균 구독 수**와 **importance 분류기 호출 빈도**. 스케일 시 importance 분류기를 변화 이벤트 *후*에만 호출하는 것이 결정적 (모든 후보를 매번 분류하면 비용 10배).

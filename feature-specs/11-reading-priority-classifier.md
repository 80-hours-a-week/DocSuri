# 11. 읽을만한 논문 분류 추천 (Reading Priority Classifier)

> 후보 논문 N편을 사용자 프로필·연구 컨텍스트와 매칭해 **Must Read / Important / Optional / Skip** 4단계로 자동 분류.

---

## 1. 핵심 요소

- **4단계 분류 정의**:
  - **Must Read**: 사용자 연구의 핵심 의존성·전제 논문. 안 읽으면 자기 연구 정당화 불가.
  - **Important**: 사용자 분야의 주류 흐름·SOTA. 읽으면 깊은 이해.
  - **Optional**: 보조·관련 분야. 시간 남으면.
  - **Skip**: 관련성 낮거나 이미 다른 논문이 더 잘 다룸.
- **다축 점수**:
  - **관련성** (사용자 프로필 임베딩 vs 논문 임베딩)
  - **영향력** (인용수, PageRank, venue tier)
  - **신선도** (출판 시기 가중치, 사용자 옵션)
  - **고유성** (다른 후보와의 중복 정도, MMR 적용)
  - **사용자 적합도** (이전 읽기·저장·번역 패턴 학습)
- **rationale 생성**: 각 분류에 *왜 이 등급인지* 1-2 문장 + anchor.
- **개인화 학습**: 사용자가 분류를 수정하면 가중치 자동 재조정 (Bandit / 간단 RL).
- **표시 한도**: 분류 결과의 시각적 균형 — Must Read 너무 많으면 의미 퇴색.

---

## 2. 주요 문제

- **분류 기준의 주관성**: 같은 논문이 사용자 A에게 Must, B에게 Optional. 사용자 컨텍스트의 정확도가 결정적.
- **Cold start**: 새 사용자는 적합도 학습 데이터 없음 → 첫 결과의 만족도 낮음.
- **"Skip"의 자기실현 위험**: Skip으로 분류된 논문을 사용자가 영원히 보지 않음 → 시스템이 놓친 진주를 영영 발굴 못함.
- **공정성**: niche 분야 논문은 인용수 낮아 자동 Optional/Skip으로 밀림. niche 사용자에겐 불리.
- **신뢰 형성**: 첫 인상이 나쁘면 시스템 신뢰 무너짐. UI에서 "왜 이 등급인지" 항상 보여야 함.
- **개인화 vs 탐험**: 너무 개인화하면 사용자가 안주, 안 하면 부정확 — exploration 강제 필요.

---

## 3. 파이프라인 설계 & 기술 스택

```
candidate 논문 N편 + user_profile
    │
    ▼
┌─────────────────────────┐
│ Feature Extractor       │
│  - 관련성 점수           │  embed cos
│  - 영향력 점수           │  citation, venue tier
│  - 신선도 점수           │  exp decay
│  - 고유성 점수 (MMR)     │
│  - 적합도 점수           │  user history → 학습된 가중치
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Multi-axis Aggregator   │  가중 합 + sigmoid
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Threshold Classifier    │  → Must/Important/Optional/Skip
│  + 균형 강제 cap         │  (Must ≤ 20%, Skip ≤ 30%)
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Rationale Generator (LLM)│  각 등급에 "왜" 설명
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Exploration Injector    │  Skip 카테고리에서 N개를         │
│                          │  "wildcard"로 Optional 승격     │
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ UI: 4-bucket layout     │
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│ Feedback Loop           │  사용자 수정 → 가중치 갱신
└─────────────────────────┘
```

### 기술 스택

| 레이어 | 선택 | 비고 |
|---|---|---|
| 임베딩 | SPECTER2 (또는 text-embedding-3-large) | |
| 인용·venue 데이터 | OpenAlex | venue tier는 자체 매핑 |
| Aggregator | sklearn linear / lightgbm | feature engineering이 핵심 |
| 가중치 학습 | Thompson Sampling (Bandit) | 단순·해석 가능 |
| Rationale LLM | Claude Haiku | 등급당 1-2 문장 충분 |
| 임계값 캐리브레이션 | per-user 분위수 정렬 | 분야별 균형 |
| UI | 4-column Kanban-style | 익숙한 UX |

---

## 4. 차별화 포인트

- **4-way 분류 + rationale**: 단순 "추천" 점수 정렬이 아닌 *명시적 행동 카테고리*. "이건 봐야 한다 vs 이건 패스해도 된다"의 결정을 시스템이 대신.
- **균형 강제 cap**: Must Read 무제한이면 무의미. 명시적 분위수 제한으로 *실제로 읽을 수 있는 양*을 만듦.
- **exploration injection**: Skip의 자기실현 위험을 의도적으로 깸. Bandit 방식으로 "어쩌면 의외의 가치"를 정기 노출.
- **anchor 부착 rationale**: "이 논문이 Must인 이유: §3의 방법이 당신의 모델 코어와 직접 충돌". 단순 추천 점수와 차별.
- **사용자 피드백 빠른 반영**: 첫 1주 만에 가중치가 개인화됨. cold start 완화.

---

## 5. 위험 요소

- **첫 인상 실패 → 이탈**: 신규 사용자의 만족도가 분야 적합도에 크게 의존. 첫 결과 후 "이 분류가 맞나요?" 캘리브레이션 단계 권장.
- **부정적 분류의 사용자 반발**: "Skip" 표시는 누군가의 논문에 부정적 라벨이 됨. 시스템 분류는 *개인 추천*임을 명시 (공개 랭킹 금지).
- **인용 편향**: 영향력 가중치가 강하면 SOTA 추종 강요. niche 연구자 소외.
- **개인화의 폐쇄 루프**: 사용자가 자기 익숙한 분야만 보게 됨. exploration 강제는 부분 해결, 사용자가 끄면 무력화.
- **모델 드리프트**: 사용자 패턴이 1년에 걸쳐 바뀌면 학습된 가중치 stale. 가중치 decay 필요.
- **LLM rationale의 비일관성**: 비슷한 두 논문이 다른 이유로 같은 등급이면 신뢰 흔들림. rationale 통일성 룰 필요.

---

## 6. 예상 비용

### 단위 비용 (1회 분류, 50 후보 논문)

| 단계 | 비용 |
|---|---|
| Feature 추출 | $0.005 (임베딩 캐시 히트) |
| Aggregator + 분류 | $0 (룰) |
| Rationale (Haiku, 50개) | ~$0.05 |
| Exploration inject | $0 |
| **합계/분류** | **~$0.06** |

> 1회당 매우 저렴. 점수·임베딩은 재사용성 높음 — 사용자가 분야 안에서 자주 호출해도 marginal cost 적음.

### MVP (사용자 1,000명 × 주 2회 분류)

- 월 8,000회 × $0.06 = **$480/월**
- 임베딩 캐시 인프라: ~$50/월
- **총 ~$530/월**

### 스케일업 (사용자 10만 명)

- 월 80만 회 × $0.06 = **$48,000/월**
- 인프라(가중치 저장·실시간 학습) ~$500/월
- **총 ~$48,500/월**

> 11개 기능 중 *단위 비용 최저* + *재사용 빈도 최고*. 사용자 lock-in 강력 (가중치 학습이 시간 갈수록 가치) — 기능 우선순위 높음.

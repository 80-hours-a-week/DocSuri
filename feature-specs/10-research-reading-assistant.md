# 10. Research Reading Assistant (논문 읽기·이해 지원)

> ⚠️ **기존 사양과의 충돌**: 원본 #11은 "논문 *쓰기* 과정 전체 지원"을 명시했으나, `AGENTS.md §1` Non-goal에 "논문 작성 보조 — 본 시스템은 소비(consumption) 도구다"가 명시되어 있다.
> 본 명세서는 이 충돌을 해소하기 위해 **"Research Reading Assistant"로 재정의**한다 — outline/draft 작성은 제외하고, "선행연구 정리·인용 후보 추천·동료 검토 시뮬레이션·아이디어 sparring"같은 소비측 보조만 다룬다.
> 작성 보조까지 진행하려면 AGENTS.md §1 개정과 시스템 정체성 재정의가 선행되어야 함.

---

## 1. 핵심 요소

- **선행연구 정리(Literature Map)**: 사용자 연구 주제를 기준으로 관련 논문 N편을 자동 그룹화 → "X 접근/Y 접근/한계점" 형식의 구조화 정리 산출.
- **인용 후보 추천**: 사용자가 쓰고 있는 문장(또는 자기 글의 청크)을 입력하면, 그 주장을 뒷받침할 적합한 *기존* 논문 후보 N편 + 인용 형식 출력 (단, *문장 생성 X*).
- **동료 검토 시뮬레이션**: 사용자 글에 대해 가상의 학회 reviewer 페르소나가 강·약점 코멘트 (rubric 기반).
- **개념 sparring**: 사용자가 "내 가설은 X다"라고 입력하면 반대 입장에 서서 도전적 질문/반례 후보 제시.
- **연구 일지 (notebook)**: 사용자가 읽은 논문·발췌·메모를 누적해 시간 흐름의 이해 변화를 추적.
- **모든 산출물에 anchor 강제**: 인용 후보·반례·약점 코멘트 모두 *근거 논문 anchor* 필수.

---

## 2. 주요 문제

- **시스템 정체성 충돌**: AGENTS.md non-goal과 표면적 충돌. "읽기 보조"로 한정해도 "쓰기 보조"로 사용자가 슬그머니 사용 가능 → 정책 강제 어려움.
- **인용 후보의 정확성**: 사용자 문장이 모호하면 LLM이 *그럴듯하지만 부적합한* 인용 후보 생성. 후보를 *반드시* anchor + abstract와 함께 제시.
- **동료 검토 시뮬레이션의 가치**: 진짜 reviewer 대비 깊이가 얕음. "체크리스트 보조" 이상으로 신뢰받기 어려움.
- **연구 일지 보안**: 사용자 진행 중인 미공개 아이디어 저장 → 보안·격리 필요.
- **학계 윤리**: 시스템 산출물을 "내가 한 것처럼" 사용하면 표절·연구부정 가능성. UI에서 명확히 "보조 도구임" 표시 + 사용 로그 권장.
- **재정의 후에도 슬립퍼리 슬로프**: "동료 검토 시뮬레이션"은 사실상 *글에 대한* 분석 → 작성 보조에 가까움. 경계 정의 필요.

## 3. 파이프라인 설계 & 기술 스택

```
┌─────────────────────────────────────────────────┐
│ 4 sub-features (사용자가 선택)                  │
│  (a) Literature Map                             │
│  (b) Citation Candidate                         │
│  (c) Peer Review Simulation                     │
│  (d) Concept Sparring                           │
└─────────────────────────────────────────────────┘
            │
    ┌───────┴────────┬─────────────┬───────────┐
    ▼                ▼             ▼           ▼

(a) Literature Map
  user topic → 06 (gap)와 05 (similar) 결합 → 구조화 정리

(b) Citation Candidate
  user sentence → embedding → ANN search →
    LLM "이 문장에 적합한 인용?" rerank → top 5 후보 + format

(c) Peer Review Simulation
  user draft → rubric(novelty/method/eval/clarity) →
    각 항목별 LLM 코멘트 + anchor

(d) Concept Sparring
  user hypothesis → corpus 내 반례/도전 후보 검색 →
    LLM 도전적 질문 + 반례 anchor

[공통] 모든 출력에 Verifier 횡단 검증
[공통] 연구 일지 적재 (사용자 옵션)
```

### 기술 스택

| 레이어 | 선택 | 비고 |
|---|---|---|
| Literature Map | 06 + 05 재사용 | 별도 LLM 호출은 합성 단계만 |
| Citation Candidate | 임베딩 ANN + Claude Sonnet rerank | |
| Peer Review | Claude Opus + rubric (NeurIPS-style) | |
| Concept Sparring | Claude Opus + 반례 검색 | 도전 톤 강조 |
| 연구 일지 | Postgres + Markdown export | 사용자 소유 강조 |
| 안전 가드 | system prompt에 "쓰기 대신 작성" 금지 명시 | |

---

## 4. 차별화 포인트

- **anchor 강제 인용 후보**: 일반 LLM이 만들어내는 가짜 인용(hallucinated DOI) 문제 없음 — 후보는 실제 검색된 논문만.
- **반대 입장 sparring**: 다수 도구가 "동의·확장"만 해준다. 본 기능은 의도적으로 반대 입장에 서서 사용자 가설을 강화시킴.
- **사용자 진화 추적**: 연구 일지에 시간순 아이디어/요약 누적 → 6개월 후 자기 변화를 회고할 수 있음.
- **"보조" 강조 UX**: 산출물에 명확히 "AI 보조" 마커 + 사용 로그 → 학계 사용 윤리 안전성.

---

## 5. 위험 요소

- **(가장 큰) 시스템 정체성 충돌**: 작성 보조 도구가 아니라는 사양과의 모순. 도입 시:
  - 옵션 A: AGENTS.md §1 개정해 시스템 미션 확장.
  - 옵션 B: 기능 #11 자체 드롭.
  - 옵션 C (본 문서 기본 가정): "Reading" 한정 재정의 + 가드레일 강제.
  - → **결정 권장**: 옵션 C로 출시 후, 사용 패턴 데이터 보고 옵션 A 결정.
- **학술 윤리/부정**: 시스템 출력을 학생이 자기 작업처럼 제출. 사용 로그 + ToS 게시 필요. 대학 통합 시 별도 정책.
- **인용 후보 부적합**: 임베딩 검색 top-5이지만 사용자 문장과 진짜 의미 일치는 다른 문제. "후보일 뿐"임을 UI에 강조.
- **Peer Review 시뮬레이션 신뢰성**: 진짜 reviewer만큼의 깊이 없음. "예비 점검" 수준 위치 부여.
- **Concept Sparring의 공격적 톤**: 사용자 멘탈 부담. 강도 조절 (옵트인).
- **연구 일지 IP**: 미공개 아이디어 저장 → 침해 시 책임. ZDR + 사용자 소유 강조.

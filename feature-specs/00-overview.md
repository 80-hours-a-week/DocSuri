## 병합/분할 판단 근거

### #4 + #9 → "04. 자동 모니터링 & 알림"
- 둘 다 **scheduler가 주기적으로 검색 쿼리를 실행해 신규 결과를 처리**하는 동일 파이프라인.
- #4는 "결과를 알린다"는 출력 측, #9는 "스케줄에 등록한다"는 입력 측을 각각 강조했을 뿐.
- 실제 구현은 `cron 등록 + diff 탐지 + 알림 디스패치`로 한 흐름.

### #6 분할
- 원문: *"연구 공백 찾아 논문 아이디어 제안 (+자신의 프로젝트에 통합할 수 있는지 분석)"*
- 두 작업의 **입력이 다름**:
  - 공백 탐지/아이디어 제안: 분야 전체 corpus만 필요.
  - 프로젝트 통합 가능성: 사용자 프로젝트 데이터 필요 (소스코드, 설계 문서).
- 산출물도 다름: 전자는 "아이디어 리스트", 후자는 "프로젝트별 액션 아이템".
- 따라서 분리.

### #6 후반 + #7 → "07. 프로젝트 트렌드 & 통합 분석"
- 둘 다 **사용자 프로젝트를 기준점으로 외부 논문과 비교**.
- #7의 "트렌드 따라가는지" + #6의 "통합 가능한지"는 동일 비교 엔진의 두 가지 질문.
- "프로젝트 컨텍스트 기반 분석 에이전트" 하나로 통합 가능.

## ⚠️ 기존 사양과의 충돌 사항

**`AGENTS.md §1` 명시 Non-goal**:
> "논문 작성 보조 (writing assistant) — 본 시스템은 **소비(consumption) 도구**다."

기능 #11 (Research Assistant)은 이 non-goal과 정면 충돌한다.
처리 옵션:
1. **드롭**: 기존 사양 유지, #11 제외.
2. **확장**: AGENTS.md 개정 후 새 미션으로 진행 (시스템 정체성 변경).
3. **재정의** (권장): "Research **Reading** Assistant"로 한정 — outline/draft 작성은 빼고 "선행연구 정리, 인용 후보 추천, 동료 검토 시뮬레이션" 같은 소비 보조 작업만 남긴다.

본 명세서는 옵션 3을 가정해 작성하되, 위험 요소 섹션에 충돌과 대안을 명시한다.

## 공통 인프라 (모든 기능이 의존)

이 11개 기능 중 대부분이 공유하는 인프라:

| 컴포넌트 | 역할 | 후보 기술 |
|---|---|---|
| Vector DB | 논문 청크 임베딩 저장/검색 | Qdrant, pgvector, Weaviate |
| Embedding model | 텍스트 → 벡터 | `text-embedding-3-large`, `voyage-3` |
| LLM | 요약/번역/분석/추론 | Claude Sonnet/Opus (prompt caching) |
| Document parser | PDF → 구조화 텍스트 | GROBID, `unstructured` |
| Job scheduler | 주기적 작업 | APScheduler, Celery beat, Temporal |
| Cache | API/임베딩 재사용 | Redis |
| External APIs | 논문 메타데이터 | Semantic Scholar, OpenAlex, arXiv, Crossref, PubMed |

> 비용 추정 시 기능별 고유 비용에 더해, 이 공통 인프라 운영비를 별도로 계산해야 한다.
